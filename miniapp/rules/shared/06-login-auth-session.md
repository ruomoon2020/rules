# 06 Login Auth Session

## 登录模型

1. `uni.login` 必须由 `auth` 或 `platform` 封装。
2. code 换 session / token 必须走后端，不得在前端拼接平台密钥。
3. token 刷新、过期、退出登录、账号切换必须有统一状态机。
4. 未登录、已登录但未绑定手机号、登录过期是不同状态，不得混用。

## 手机号与用户资料

1. 手机号授权必须由用户动作触发。
2. 授权失败、用户拒绝、接口不可用必须有降级提示。
3. 不得把手机号作为登录态唯一凭据；以后端 session / token 为准。
4. 用户头像昵称等资料不得默认视为实名信息。

## 业务权限边界

1. C 端小程序默认只做登录态、手机号绑定、会员状态等轻量状态；**不默认复制管理端的菜单、角色、按钮权限体系**。
2. 是否能下单、退款、改价、查看他人数据、导出等高风险能力，以后端鉴权和数据权限为准；前端隐藏入口只用于体验优化。
3. B 端 / 店员 / 导购 / 仓管类小程序可消费后端返回的 `roles` / `permissions[]` / ability bits，但小程序不得维护角色、菜单、权限主数据。
4. 小程序 `hasPermission()`（若有）只能控制入口与按钮展示，禁止作为安全来源；接口仍须返回 403 / `NO_PERMISSION` 等稳定错误。
5. 与管理端共用权限码时，小程序只消费子集；权限码来源须能追溯到 OpenAPI / 后端枚举，禁止页面临时编造。

## 存储

1. token 不得明文长期存储；按项目安全方案选择短期 storage 或内存态。
2. 敏感用户信息不得散落在多个 store 和 storage key。
3. logout 必须清理 auth store、storage、缓存和待重试队列。
