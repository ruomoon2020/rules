一、 项目结构规范
结合 Monorepo 与 Feature-Sliced Design (FSD) 思路，确保高内聚低耦合。

text
├── apps/                    # 应用入口 (多应用隔离)
│   ├── web/                 # 主站点
│   └── admin/               # 后台管理系统
├── packages/                # 共享库
│   ├── shared/              # 纯工具、UI库、通用类型
│   └── features/            # 独立业务功能 (如: checkout, comments)
├── configs/                 # 统一配置 (ESLint, TSConfig, Prettier)
└── tools/                   # 自定义脚本
核心规则：

严禁反向依赖：apps 可依赖 packages，但 packages 之间禁止循环引用。

就近放置：页面独用的组件，放在该页面的 components 文件夹下，而非全局 components。

路径别名：使用 @/ 代表 src，@shared 代表共享库，禁用 ../../../ 这种超过两级的长路径。

三、 代码编码规则 (ESLint + Prettier)
以 Airbnb风格 为基底，结合 Google 规则。

1. 基础语法
变量定义：强制使用 const 或 let，禁止使用 var。

分号：必须加分号。Google 和标准 ESLint 配置均强制要求，以避免 ASI（自动分号插入）导致的问题。

引号：统一使用 单引号 (' ')。

逗号：多行对象或数组结尾，必须加逗号 (trailing comma)，便于 git diff 追踪。

2. 代码逻辑
相等判断：使用 === 和 !==，严禁使用 ==。

早期返回：优先处理错误边界和边缘情况，减少 if...else 嵌套。

TypeScript 优先：禁止使用 any。未知类型请用 unknown 配合类型守卫。

五、 变量与方法命名规则
核心原则：顾名思义，没有歧义。

1. 变量命名
普通变量/常量：名词 + 限定词。userName (好) vs data (差)。

布尔值：使用 is、has、can、should 前缀。isLoading, hasPermission。

数字：使用 max, min, total 前缀。maxRetryCount, totalPrice。

集合 (Array/Map)：使用复数形式。users, orderMap。避免 userList 这种匈牙利命名法。

2. 方法/函数命名
动词开头 + 具体意图：

获取/查询：get, fetch, query (异步用 fetch 或 useQuery)

创建/更新：create, update, save

移除：delete, remove

事件处理：handle + 元素 + 动作（handleSubmitClick）

回调函数：参数名使用 on + 名词。onSuccess, onError。

3. 类/组件命名
组件：名词或名词短语，体现功能。LoginForm, DataGrid。

服务 (Service)：名词 + Service。UserService, AnalyticsService。

工具类：名词 + 动词ing 或名词。Formatter, Validator。

六、 日志与异常处理规则
参考 阿里规范 与 谷歌 Closure 库。

1. 日志分级
console.error：仅用于开发环境。生产环境必须移除或替换为上报系统。

logger.info：关键用户行为追踪（如：提交订单、支付发起）。

logger.warn：非致命性错误，如接口降级、权限不足。

2. 错误处理与返回
遵循阿里 API 规约，前端全局拦截器需将后端返回转换为统一格式：

typescript
interface ApiResponse<T> {
  code: string | number;      // 业务状态码 ('1001', '401')
  message: string;            // 后端调试信息
  data: T | null;
  success: boolean;
}
处理策略：

code === '200'：正常返回 data。

code === '401'：清空缓存，跳转登录页。

code === '403'：弹出“无权限”提示。

其他 code：使用统一 Toast 组件展示 message。

七、 样式与 UI 规范 (CSS/SCSS)
结合 BEM 命名 与 Design Token。

1. 命名规则
Block (块)：组件名。.user-card

Element (元素)：__ 连接。.user-card__avatar

Modifier (修饰符)：-- 连接。.user-card--highlighted

响应式单位：全局使用 rem（基于 16px 基准），边框使用 px。

2. 页面经典布局参数

页面主体内容：左右边距 24px，顶部边距 20px。

卡片内边距：通常使用 16px 或 20px。

表单项间距：底部边距 16px。

使用 CSS 变量：--color-primary, --shadow-sm, --radius-md。

八、 组件与 UI 元素规范
1. 按钮 (Button) 统一风格
类型 (Variant)：

主按钮 (Primary)：品牌色填充，用于“提交”、“保存”、“确认”等核心操作（页面只能有一个 Primary 按钮）。

次按钮 (Secondary/Default)：灰色边框填充，用于“取消”、“返回”。

文字按钮 (Text)：纯文字，用于“忘记密码？”等辅助操作。

危险按钮 (Danger)：红色，用于“删除”、“注销”。

尺寸 (Size)：

大 (Large)：高度 40px，用于重要页面。

中 (Medium)：高度 32px (默认)。

小 (Small)：高度 24px，用于表格内。

交互：加载中状态（Loading）必须支持，禁用时不可点击。

2. 弹出框 (Modal / Drawer)
确认弹窗 (Confirm Modal)：

标题：明确提问（如：“删除这条评论？”）。

内容：解释后果（如：“此操作不可逆。”）。

按钮顺序：右键为“确定”（Primary），左键为“取消”（Default）。

抽屉 (Drawer)：

用于表单详情或复杂编辑。

宽度建议：400px (简单表单)、680px (标准)、800px+ (复杂)。

必须支持点击遮罩层关闭。

九、 配置管理规范
遵循 十二要素应用 原则：

环境隔离：.env.development, .env.production, .env.test。

命名前缀：VITE_ 或 REACT_APP_ 开头，区分客户端变量。

禁止硬编码：API 地址、密钥、静态资源 CDN 前缀必须通过环境变量注入。

单一事实来源：

prettierrc、eslintrc、tsconfig.json 放在根目录或 configs/ 包下统一管理。

十、 前端错误监控规则
错误边界 (Error Boundary)：

React/Vue 应用中，必须在根组件外包裹一层错误边界 UI，防止白屏。

降级 UI 应包含：“出错了” + “刷新页面” 按钮。

脚本错误上报：

监听 window.onerror 和 unhandledrejection。

提取错误堆栈，上报至 Sentry 或内部监控平台，绝不只 console.log。
