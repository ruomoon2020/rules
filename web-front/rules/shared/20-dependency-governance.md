# Dependency Governance Rules

用于新增、升级、替换 npm 依赖和大型运行时库。

## 新增依赖前

1. 先确认项目已有能力：工具函数、Base 组件、内部包、浏览器原生 API。
2. PR 必须说明引入原因、使用范围、替代方案、预估体积、维护状态和许可证。
3. 禁止为单个页面的小需求引入重型通用库。
4. 禁止引入功能重复依赖，例如同项目同时存在多个日期库、请求库、图表库。
5. 禁止引入无人维护、周下载/社区活跃明显异常、长期未修高危漏洞的包。

## 版本与锁定

1. 使用项目统一包管理器和 lockfile；不得手改 lockfile 规避冲突。
2. 依赖升级须说明 breaking changes 与回归范围。
3. 安全修复升级优先，但仍须跑测试与构建。
4. 不得在业务代码中依赖未声明的 transitive dependency。

## 体积与加载

1. 新依赖进入首屏 bundle 前必须评估体积。
2. 图表、地图、富文本、PDF、Office 预览、代码编辑器等重型依赖默认懒加载。
3. 工具库优先按需 import；禁止整包引入导致 tree-shaking 失效。
4. 超出 `07-security-performance.md` 预算时，须给出拆包、懒加载或豁免依据。

## 安全与许可证

1. 新依赖须通过项目 audit / SCA / license check。
2. 禁止引入许可证与项目分发方式冲突的包。
3. 处理文件、HTML、URL、加密、压缩、解析类依赖时，须额外关注安全记录。
4. 禁止在前端依赖中打包密钥、私有证书、服务端 SDK 凭证。

## AI 约束

- AI 不得直接修改 `package.json` 添加依赖，除非用户明确要求且已说明原因。
- AI 推荐依赖时必须给出“不新增依赖”的替代方案。
- 不确定项目已有依赖时，先读 `package.json` 和 lockfile。
