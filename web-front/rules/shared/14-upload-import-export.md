# 上传、导入与导出规则

适用于上传、模板下载、Excel / CSV / JSON / Word 导入导出。执行配合 `05-api-contract.md`、`06-state-route-permission.md`、`18-logging-observability.md` 与业务仓上传组件；具体解析库、下载 API、任务轮询 API 以目标仓库为准，AI 不得虚构方法签名。

## 上传

1. 限制文件类型、大小、数量；错误提示必须可理解。
2. 上传过程展示进度；支持取消（若产品要求）。
3. 大文件按项目方案走分片 / 直传，不在页面硬编码密钥、bucket、AK/SK、临时凭证。
4. 上传前后端都必须校验类型与大小；前端校验只做体验，不作为安全边界。
5. 预览文件前须确认来源可信；富文本 / HTML 内容必须走安全封装。

## 导入通用流程

1. 导入必须提供模板下载；模板包含字段名、字段 key、必填、类型、枚举、示例与说明。
2. 导入字段必须来自 schema / generated 类型；禁止导入 schema 中不存在的字段。
3. 导入前预校验：文件类型、大小、编码、sheet、表头、必填列、字段类型、枚举值、日期格式。
4. 导入结果必须区分成功 / 部分失败 / 全部失败；部分失败须展示成功数、失败数，并支持下载错误明细。
5. 错误明细必须包含行号、列名、字段 key、原始值摘要、错误原因；禁止只返回“格式错误”。
6. 长任务必须走异步任务：提交任务 → 展示任务 ID / 进度 → 轮询或通知 → 可取消 / 可重试。
7. 防重复提交：导入按钮 loading；同一文件重复导入按业务规则提示或阻断。
8. 导入任务必须具备幂等标识（如 `taskId` / `idempotencyKey` / 文件 hash + 业务 key），避免刷新、重试、网络抖动造成重复入库。
9. 高风险导入（用户、角色、权限、金额、库存、配置、批量删除 / 覆盖类数据）须先预校验并展示影响摘要，用户二次确认后再正式提交。

## Excel 导入导出

1. 支持 `.xlsx` 优先；如支持 `.xls`，须明确兼容范围。
2. 表头匹配优先使用字段 key 或模板定义，不要只依赖列顺序。
3. 多 sheet 导入须明确 sheet 名与结构；禁止默认读取第一个 sheet 后静默忽略其余 sheet。
4. 日期、金额、百分比、枚举必须按模板格式解析；未知枚举值不得静默吞掉。
5. 导出 Excel 的列顺序、列名、枚举展示值须与当前业务视图或导出模板一致。

## CSV 导入导出

1. 明确编码（默认 UTF-8，可按业务要求带 BOM）、分隔符、换行符。
2. CSV 导入须处理引号、逗号、换行、空值；不要用简单 `split(',')` 解析业务 CSV。
3. CSV / Excel 导出必须防公式注入：以 `=`, `+`, `-`, `@` 开头的单元格按项目策略转义或前置 `'`。
4. CSV 导出含中文时须按项目约定处理 BOM，避免 Excel 打开乱码。

## 日期、数字与国际化格式

1. 导入导出必须明确时区、日期格式、时间精度（秒 / 毫秒）与空值含义；禁止让浏览器或 Excel 自动猜日期。
2. 金额、百分比、数量须明确小数位、舍入规则、货币单位、千分位与负数格式。
3. 跨区域项目须按项目 locale 约定处理小数点、分隔符和日期展示；导入模板中必须写明格式示例。

## JSON 导入导出

1. JSON 导入必须使用 schema / parser 校验结构；禁止任意 JSON 透传到 API。
2. 导入前限制文件大小与对象层级，避免超大 JSON 阻塞页面。
3. JSON 导出须明确结构版本，如 `version` / `schemaVersion`；破坏性结构变更须兼容旧版本或提供迁移说明。
4. JSON 中的敏感字段按权限脱敏；禁止导出 Token、密钥、内部调试字段。

## Word 导入导出

1. 先明确场景：`.docx` 模板填充、富文本导出、还是文档解析导入；三者不要混用。
2. Word 导出模板变量必须来自 schema / generated 类型或明确的 view model；禁止拼接未经消毒的 HTML。
3. 富文本导出 Word 前必须走安全 HTML 消毒；图片、附件、外链按项目安全策略处理。
4. Word 导入只解析项目明确支持的字段；无法可靠解析的内容须提示人工处理，不得猜测字段。

## 导出通用规则

1. 大数据量走异步导出 + 通知 / 下载中心，不阻塞页面。
2. 导出字段与权限、schema 可见字段一致；导出条件应与当前筛选条件一致并可追溯。
3. 异步导出须保存提交时的筛选条件快照；用户后续修改筛选条件不得影响已提交导出任务。
4. 文件名含业务含义、时间戳，必要时含筛选摘要；避免泄露敏感条件。
5. 下载链接必须走鉴权；临时链接应有有效期或一次性 token，禁止把永久公开 URL 暴露给前端长期缓存。
6. 错误明细、导出文件、模板文件若含租户或权限边界数据，下载时必须再次校验权限。
7. 敏感字段默认脱敏；只有具备明确权限时才允许明文导出。
8. 导出失败须有可重试入口；异步导出须能查看任务状态。

## 权限、审计与日志

1. 模板下载、导入、导出、错误报告下载应分别使用权限码或统一权限 helper。
2. 导入 / 导出必须有审计记录：操作人、时间、模块、条件摘要、文件名、任务 ID、结果。与后端字段映射见 monorepo `web-backend/rules/docs/fullstack-contract.md` §审计日志（`operatorId`、`occurredAt`、`resourceType`、`action`、`requestSummary`、`result`、`traceId` 等）。
3. 错误报告本身也可能包含敏感数据，下载错误报告须鉴权并按权限脱敏。
4. 日志与埋点遵守 `18-logging-observability.md`；不得记录完整文件内容、完整请求体或敏感字段。

## 禁止

- 在生产代码留测试用上传接口或假进度。
- 导出未授权字段、隐藏字段、内部调试字段。
- 前端私自新增导入字段绕过 schema / generated。
- 用字符串拼接或简单 split 解析复杂 CSV / Excel。
- 导入失败只显示“失败”而无行列级错误明细。
- 使用永久公开下载 URL 承载敏感导出文件。
- 高风险导入无预览确认、无幂等标识。
