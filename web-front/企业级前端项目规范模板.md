# 1. 规范目标与适用范围

## 1.1 规范目标

本规范用于统一企业级前端项目的架构、目录、编码、组件、UI、交互、错误处理、权限、安全、测试和 AI 编码行为，解决以下问题：

1. 页面风格不统一。
2. 按钮、弹窗、表格、筛选、表单样式不一致。
3. 业务页面重复造轮子。
4. API、错误处理、权限控制分散。
5. 变量、方法、文件命名随意。
6. 项目目录随业务增长逐渐混乱。
7. AI 工具生成代码缺少约束。
8. 新人接手成本高，Code Review 口径不统一。

## 1.2 适用范围

适用于以下项目：

- Vue 3 企业后台管理系统。
- Vue 3 + Element Plus 管理端项目。
- Vite + TypeScript 工程。
- 需要使用 Codex、Cursor、Copilot 等 AI 工具辅助开发的项目。
- 需要沉淀企业级设计系统和公共组件体系的项目。

## 1.3 规范分层

| 层级 | 内容 | 目标 |
|---|---|---|
| 设计规范层 | 颜色、字体、间距、主题、布局、组件视觉 | 保证界面一致性 |
| 组件规范层 | 按钮、弹窗、表格、筛选、表单、上传、详情页 | 保证组件复用 |
| 代码规范层 | 目录、命名、接口、状态、权限、错误处理 | 保证代码可维护 |
| 工程规范层 | Lint、格式化、测试、构建、CI、Code Review | 保证交付质量 |
| AI 规则层 | AGENTS.md、Cursor Rules、Context Pinning、AST Lint、Schema SSOT | 保证 AI 输出可控且可机检 |

## 1.4 规范优先级与冲突处理

当需求与规范冲突时，按以下优先级执行（高 → 低）：

1. **安全与合规**（§30、§37.4、后端鉴权）。
2. **本规范正文**（经团队评审的例外须在 PR 中说明）。
3. **项目待完善清单 §46** 中已勾选确认的项目定制项。
4. **临时实现**（须创建跟进任务，不得长期遗留）。

## 1.5 术语表

| 术语 | 含义 |
|---|---|
| Base 组件 | `components/base` 下对 Element Plus 的统一封装 |
| 壳层 | Sidebar + Header + TagsView + Main 的应用框架 |
| 字典 | 后端下发的可配置枚举（如状态、类型），非前端硬编码 |
| DataScope | 数据行级可见范围（本人/本部门/全部等） |
| CodeGen | 由 OpenAPI 自动生成 API 类型与 Client |
| TagsView | 多页签栏，与 `keep-alive` 配合 |
| Context Pinning | AI 编码前强制阅读 Base 组件定义，防止虚构 Props |
| Monorepo | pnpm workspace + Turborepo，多应用共享 packages |
| Schema SSOT | 根目录 `contracts/schema.json`（或等价）为前后端结构体唯一真相源，驱动 CodeGen 与 AI |
| AST Lint | 通过 ESLint 在 `views/` 等目录禁止引入/使用原生 `el-*`，保存即报错 |

# 2. 前端架构总则

## 2.1 架构原则

| 原则 | 说明 |
|---|---|
| 统一入口 | API、权限、弹窗、按钮、主题、错误处理必须有统一入口 |
| 组件优先 | 业务页面不得重复造基础 UI 组件 |
| 配置优先 | 页面可配置项应配置化，避免硬编码 |
| 类型优先 | TypeScript 类型先行，接口字段不允许随意使用 |
| 可维护优先 | 优先清晰结构，不追求短期堆页面速度 |
| 可替换优先 | UI、请求层、主题应保持低耦合 |
| 可审计优先 | 操作、错误、权限、日志应可追踪 |
| AI 可控 | AI 只能按项目规则生成代码，不能自由发挥 |

## 2.2 分层结构

```text
视图层 views
  ↓
业务组件层 components/business
  ↓
基础组件层 components/base
  ↓
组合逻辑层 composables / hooks
  ↓
状态层 store
  ↓
接口层 api
  ↓
工具层 utils
  ↓
类型层 types / enums / constants
```

## 2.3 依赖方向

允许：

```text
views -> business components -> base components
views -> hooks -> api
views -> store
api -> request wrapper
components -> types / constants / enums
```

禁止：

```text
base components -> views
base components -> specific business API
utils -> Vue component instance
store -> DOM operation
api -> Element Plus message directly
```

## 2.4 页面职责

| 层级 | 允许 | 禁止 |
|---|---|---|
| views | 页面编排、业务组合、事件转发 | 大量公共逻辑、重复封装表格/弹窗 |
| components/base | 通用 UI 能力 | 依赖业务接口和业务字段 |
| components/business | 可复用业务组件 | 放一次性页面代码 |
| hooks/composables | 可复用逻辑 | 直接写样式或强依赖 DOM |
| api | 接口声明、请求封装 | 弹窗、消息提示、页面状态 |
| store | 用户、权限、主题、字典等全局状态 | 页面私有状态泛滥 |
| utils | 纯函数工具 | 引用组件实例、路由实例乱用 |

## 2.5 扩展模式（按需）

| 模式 | 说明 |
|---|---|
| **Monorepo 多项目演进** | 团队内出现 **2 个及以上**关联系统（主控后台、商户端、大屏、移动 H5 等）时，**必须**转为 `pnpm workspace` + **Turborepo**，避免多套 `components/base`、`utils` 多头维护 |
| 多租户 | 请求头或子域名携带 `tenantId`，主题/Logo 可按租户配置 |
| 微前端 | qiankun / wujie：子应用独立构建，壳层负责菜单与认证；Token、权限、主题由主应用下发 |
| 低代码 | 列表/表单预留 Schema，与 `BaseTable` / `BaseForm` 字段配置对齐 |

未启用时不必引入相关依赖。

### 2.5.1 Monorepo 目录与包职责（≥2 个关联系统时）

```text
repo/
├── apps/
│   ├── admin-web/              # 主控后台
│   ├── merchant-web/           # 商户后台
│   └── mobile-h5/              # 移动端 H5（可选）
├── packages/
│   ├── components/             # 原 components/base + layout 公共 UI
│   ├── shared/                 # utils/request、format、security、types、constants
│   ├── api-client/             # 可选：OpenAPI 生成 + 薄封装
│   └── eslint-config/          # 可选：统一 ESLint 预设
├── pnpm-workspace.yaml
├── turbo.json
└── package.json
```

| 包 | 职责 |
|---|---|
| `@corp/components` | `BaseTable`、`BaseForm` 等，**禁止**依赖具体业务 API |
| `@corp/shared` | request、format、security、logger |
| `apps/*` | 仅放 `views`、业务 `features`、应用级路由与 store |

依赖方向：`apps → packages`，**禁止** `packages → apps`。子应用通过 workspace 协议引用：`"@corp/components": "workspace:*"`。

## 3.2 目录职责

| 目录 | 说明 |
|---|---|
| `api` | 所有接口请求方法，不允许组件中直接写 axios |
| `assets` | 图片、图标、字体等静态资源 |
| `components/base` | 基础通用组件，与业务无关 |
| `components/business` | 跨页面复用的业务组件 |
| `composables` / `hooks` | 组合式复用逻辑 |
| `constants` | 常量定义 |
| `directives` | 自定义指令，如权限指令 |
| `enums` | 枚举定义 |
| `layouts` | 页面整体布局 |
| `plugins` | 第三方插件注册 |
| `router` | 路由配置和守卫 |
| `store` | 全局状态 |
| `styles` | 全局样式、主题、设计 token |
| `types` | 全局类型、业务类型 |
| `utils` | 纯工具函数；含 `request/`、`format/`、`security/` |
| `monitor/` | 错误与性能上报（可与 utils 合并，团队择一） |
| `views` | 页面级组件 |
| `public/` | favicon、静态页；构建时原样拷贝 |

## 3.3 Monorepo 下的目录映射（见 §2.5.1）

单仓 `src/components/base` → `packages/components`；`src/utils` → `packages/shared`。各 `apps/*` 仅保留 `views/`、业务 store、应用路由。

# 4. 技术栈规范

## 4.1 标准技术栈

| 类别 | 推荐技术 |
|---|---|
| 框架 | Vue 3 |
| 语言 | TypeScript |
| 构建工具 | Vite |
| UI 组件库 | Element Plus |
| 路由 | Vue Router |
| 状态管理 | Pinia 或项目现有 store 方案 |
| 请求库 | Axios 二次封装 |
| 样式 | SCSS + CSS Variables |
| 国际化 | vue-i18n |
| 图标 | 统一图标库 |
| 单元测试 | Vitest + Vue Test Utils |
| E2E 测试 | Playwright |
| 代码检查 | ESLint + Prettier + Stylelint |
| 提交规范 | commitlint + husky |

## 4.2 依赖引入原则

1. 不得随意引入新 UI 组件库。
2. 不得为了单个页面引入大型依赖。
3. 新增依赖必须说明用途、体积、维护状态和替代方案。
4. 所有工具库应集中封装后使用。
5. 生产依赖和开发依赖必须区分清楚。

## 4.3 环境与版本要求

| 项目 | 建议约定 |
|---|---|
| Node.js | 使用 LTS（如 20.x / 22.x），在 `.nvmrc` 或 `package.json#engines` 中锁定 |
| 包管理器 | 团队统一使用 pnpm（推荐）或 npm，禁止混用 lock 文件 |
| 浏览器 | 明确最低版本（如 Chrome 90+、Edge 90+），写入 README |
| TypeScript | 开启 `strict`，提交前必须通过 `vue-tsc --noEmit` |

环境变量文件约定：

```text
.env                # 本地默认（不提交密钥）
.env.development    # 开发环境
.env.test           # 测试环境
.env.production     # 生产构建
```

| 变量前缀 | 说明 |
|---|---|
| `VITE_` | 仅暴露给前端的变量（Vite 约定） |
| 禁止提交 | 密钥、内网地址、私有 Token 不得写入仓库 |

常用变量示例：

```text
VITE_APP_TITLE=企业管理系统
VITE_API_BASE_URL=https://api.example.com
VITE_UPLOAD_MAX_SIZE=20971520
```

## 4.4 Vite 工程约定

`vite.config.ts` 建议统一约定：

| 项 | 约定 |
|---|---|
| `resolve.alias` | `@` → `src/` |
| `server.proxy` | 开发环境代理 `/api` 到后端，禁止生产写死内网地址 |
| `build.rollupOptions.output.manualChunks` | 拆分 `vue`、`element-plus`、`echarts` 等大库 |
| 按需引入 | `unplugin-vue-components` + `unplugin-auto-import` 解析 Element Plus |
| 环境校验 | 构建前脚本检查必填 `VITE_*`（可选 `vite-plugin-env-compatible`） |
| 体积分析 | **必须** 配置 `rollup-plugin-visualizer`，`pnpm build:analyze` 产出 `stats.html` |
| 体积门禁 | CI 检查单 chunk（**gzip 前**）不超过 **500KB**，超出 Warning；超过 **800KB** 可配置为 Error 阻塞合并 |

```ts
// 分包示例（按项目调整）
manualChunks: {
  vue: ['vue', 'vue-router', 'pinia'],
  'element-plus': ['element-plus'],
  echarts: ['echarts'], // 图表单独拆包
}
```

`package.json` 脚本示例：

```json
{
  "scripts": {
    "build": "vite build",
    "build:analyze": "vite build && open dist/stats.html"
  }
}
```

禁止在业务代码中绕过 alias 使用超长相对路径 `../../../`。

# 5. 设计 Token 规范

## 5.1 Token 分层

```text
Primitive Token     原始值，如 blue-6、gray-1
Semantic Token      语义值，如 primary、danger、bg-page
Component Token     组件值，如 button-primary-bg、dialog-header-bg
```

## 5.2 色彩 Token

```scss
:root {
  /* Brand */
  --app-color-primary: #1677ff;
  --app-color-primary-hover: #4096ff;
  --app-color-primary-active: #0958d9;

  /* Functional */
  --app-color-success: #52c41a;
  --app-color-warning: #faad14;
  --app-color-danger: #ff4d4f;
  --app-color-info: #909399;

  /* Text */
  --app-color-text-primary: #1f2329;
  --app-color-text-regular: #4e5969;
  --app-color-text-secondary: #86909c;
  --app-color-text-disabled: #c9cdd4;

  /* Background */
  --app-color-bg-page: #f5f7fa;
  --app-color-bg-container: #ffffff;
  --app-color-bg-muted: #f2f3f5;

  /* Border */
  --app-color-border-base: #dcdfe6;
  --app-color-border-light: #e5e6eb;
  --app-color-border-lighter: #ebeef5;
}
```

## 5.3 间距 Token

```scss
:root {
  --app-spacing-xxs: 4px;
  --app-spacing-xs: 8px;
  --app-spacing-sm: 12px;
  --app-spacing-md: 16px;
  --app-spacing-lg: 24px;
  --app-spacing-xl: 32px;
  --app-spacing-xxl: 48px;
}
```

## 5.4 字体 Token

```scss
:root {
  --app-font-size-xs: 12px;
  --app-font-size-sm: 13px;
  --app-font-size-base: 14px;
  --app-font-size-md: 16px;
  --app-font-size-lg: 18px;
  --app-font-size-xl: 20px;
  --app-font-size-title: 24px;

  --app-line-height-base: 1.5;
  --app-font-weight-regular: 400;
  --app-font-weight-medium: 500;
  --app-font-weight-bold: 600;
}
```

## 5.5 圆角 Token

```scss
:root {
  --app-radius-xs: 2px;
  --app-radius-sm: 4px;
  --app-radius-base: 6px;
  --app-radius-md: 8px;
  --app-radius-lg: 12px;
  --app-radius-full: 999px;
}
```

## 5.6 阴影 Token

```scss
:root {
  --app-shadow-sm: 0 1px 2px rgb(0 0 0 / 4%);
  --app-shadow-base: 0 2px 8px rgb(0 0 0 / 8%);
  --app-shadow-lg: 0 8px 24px rgb(0 0 0 / 12%);
}
```

## 5.7 Token 使用规则

1. 业务组件不得直接写主题色十六进制值。
2. 业务组件必须使用语义化 Token。
3. 基础组件可使用 Component Token。
4. 设计 Token 修改必须经过视觉回归验证。
5. Token 命名必须体现用途，不只体现颜色。

## 5.8 层级 Token（z-index）

```scss
:root {
  --app-z-dropdown: 1000;
  --app-z-sticky: 1020;
  --app-z-fixed: 1030;
  --app-z-modal-backdrop: 1040;
  --app-z-modal: 1050;
  --app-z-popover: 1060;
  --app-z-tooltip: 1070;
  --app-z-toast: 1080;
}
```

业务组件禁止随意写 `z-index: 9999`，必须使用上表刻度。

## 5.9 动效 Token

```scss
:root {
  --app-motion-fast: 150ms;
  --app-motion-base: 250ms;
  --app-motion-slow: 350ms;
  --app-ease-out: cubic-bezier(0.215, 0.61, 0.355, 1);
  --app-ease-in-out: cubic-bezier(0.645, 0.045, 0.355, 1);
}
```

需尊重系统偏好：在 `@media (prefers-reduced-motion: reduce)` 下缩短或关闭非必要动画。

## 5.10 断点 Token

```scss
:root {
  --app-breakpoint-sm: 768px;
  --app-breakpoint-md: 992px;
  --app-breakpoint-lg: 1200px;
  --app-breakpoint-xl: 1600px;
}
```

后台以桌面为主，但筛选区、工具栏在 `sm` 以下应可折叠或换行，避免横向溢出。

## 5.11 密度模式

建议支持 `comfortable`（默认）与 `compact` 两档，通过根节点 `data-density` 切换表格行高、表单项间距，并与 Element Plus `ConfigProvider#size` 联动。

## 5.12 设计交付（Figma）

1. Figma Variables 与 `--app-*` Token 建立映射表（颜色、间距、圆角）。
2. 组件交付须含：默认 / hover / disabled / loading、暗色稿（若启用）。
3. 视觉变更须同步更新 Token 并走 Code Review + 可选视觉回归。

# 6. 主题规范

## 6.1 主题类型

建议至少支持：

| 主题 | 场景 |
|---|---|
| `light` | 默认浅色主题 |
| `dark` | 深色主题 |
| `blue` | 政企蓝主题 |
| `green` | 环保、能源、监管类主题 |

## 6.2 主题切换原则

1. 业务代码不得直接判断具体主题。
2. 业务组件只使用 CSS Variables。
3. 主题切换通过根节点 `data-theme` 或 class 实现。
4. 不允许在业务组件中写 `theme === 'dark'` 之类判断。

示例：

```html
<html data-theme="light">
```

```scss
:root[data-theme='light'] {
  --app-color-bg-page: #f5f7fa;
  --app-color-bg-container: #ffffff;
  --app-color-text-primary: #1f2329;
}

:root[data-theme='dark'] {
  --app-color-bg-page: #141414;
  --app-color-bg-container: #1f1f1f;
  --app-color-text-primary: #f5f5f5;
}
```

# 7. 页面布局规范

## 7.1 标准后台管理页面结构

```text
BasePage
├─ BasePageHeader
├─ BaseFilterBar
├─ BaseToolbar
├─ BaseTable
├─ BasePagination
└─ Dialog / Drawer
```

## 7.2 页面类型

| 页面类型 | 适用场景 |
|---|---|
| 标准列表页 | 管理类页面，如用户、角色、字典 |
| 左树右表 | 组织、部门、分类、区域 |
| 表单页 | 新增、编辑、配置 |
| 详情页 | 查看业务对象详情 |
| Dashboard | 首页、监控、统计分析 |
| 地图页 | GIS、物联网、监管平台 |
| 流程页 | 审批、向导、分步配置 |

## 7.3 页面布局禁止事项

1. 禁止每个页面单独写布局容器。
2. 禁止页面背景色不一致。
3. 禁止随意设置大面积固定高度。
4. 禁止筛选、工具栏、表格、分页位置混乱。
5. 禁止页面底部出现大量无意义空白。

## 7.4 应用壳层结构

标准后台应用壳层：

```text
AppLayout
├─ AppSidebar          # 侧栏菜单
├─ AppHeader           # 顶栏（折叠、面包屑、用户区）
├─ AppTagsView         # 多页签（可选）
├─ AppMain             # 路由出口 + keep-alive
└─ AppSettings         # 主题/密度/语言（可选抽屉）
```

1. 菜单数据由权限路由生成，禁止手写重复菜单树。
2. 顶栏与侧栏使用统一 Token，不得各页面自定义壳层样式。
3. 内容区仅渲染 `router-view`，业务页面使用 `BasePage` 而非再套一层全屏布局。

## 7.5 登录 / 注册 / 找回密码页

| 项 | 规范 |
|---|---|
| 布局 | 居中卡片或左右分栏；使用 `BasePage` 或独立 `LoginLayout` |
| 表单 | 账号、密码、验证码（若启用）；提交 loading、防重复点击 |
| 校验 | 前端格式校验 + 后端错误映射到字段 |
| SSO | OAuth/OIDC 跳转统一入口，回调后写入 Token 并拉权限 |
| 安全 | 禁止 URL 带密码；登录失败不提示「用户是否存在」等敏感信息 |
| 会话 | 记住我仅延长 Refresh 周期，不长期明文存密码 |

## 7.6 Dashboard / 图表页

1. 图表库（如 ECharts）路由级懒加载，主题色读取 `--app-color-*`。
2. 容器须 `resize` 监听（`ResizeObserver` 或封装 hook），避免侧栏折叠后图表变形。
3. 无数据用 `BaseEmpty`，加载用 `BaseSkeleton`。
4. 自动刷新须可关闭，间隔不少于 30s；离开页面清除定时器。
5. 大屏/驾驶舱可单独布局，但仍须走 Token，禁止满屏硬编码色值。

# 8. 页头规范

## 8.1 页头内容

标准页头可包含：

- 标题 `title`
- 副标题 `description`
- 面包屑 `breadcrumb`
- 返回按钮 `back`
- 状态标签 `status`
- 右侧操作 `actions`

## 8.2 页头规则

1. 标题必须清晰表达当前页面业务对象。
2. 标题不超过 12 个中文字符，复杂场景用副标题补充。
3. 面包屑不超过 4 级。
4. 返回按钮只用于详情页、编辑页、流程页。
5. 页头右侧操作不超过 3 个，超过放入“更多”。
6. 页头不得直接承载复杂业务表单。

# 11. 按钮规范

## 11.1 按钮类型

| 类型 | 用途 |
|---|---|
| Primary | 新增、保存、提交、查询等主操作 |
| Default | 取消、返回、重置等次操作 |
| Success | 启用、通过、完成 |
| Warning | 暂停、提醒、待确认 |
| Danger | 删除、禁用、撤销 |
| Text | 表格行内操作 |
| Link | 跳转类操作 |
| Icon | 纯图标操作 |
| Dropdown | 更多操作 |

## 11.2 按钮语义映射

| 操作 | 按钮类型 |
|---|---|
| 新增 | Primary |
| 保存 | Primary |
| 提交 | Primary |
| 查询 | Primary |
| 重置 | Default |
| 取消 | Default |
| 返回 | Default |
| 删除 | Danger |
| 批量删除 | Danger |
| 导入 | Default |
| 导出 | Default |
| 启用 | Success |
| 禁用 | Warning / Danger |
| 查看 | Text |
| 编辑 | Text |
| 更多 | Dropdown |

## 11.3 按钮尺寸

| 尺寸 | 高度 | 场景 |
|---|---|---|
| small | 28px | 表格行内、紧凑布局 |
| default | 32px | 普通页面 |
| large | 40px | 登录页、重要表单 |

## 11.4 按钮数量限制

1. 页面工具栏主按钮不超过 5 个。
2. 表格行内操作不超过 3 个。
3. 超过 3 个操作必须放入“更多”。
4. 危险操作不能和主按钮放得过近。
5. 删除按钮必须有二次确认。

## 11.5 按钮状态

所有按钮必须考虑以下状态：

- default
- hover
- active
- focus
- disabled
- loading
- permission-hidden
- permission-disabled

## 11.6 禁止事项

1. 禁止业务页面直接使用 `el-button` 写复杂样式。
2. 禁止 hardcode 按钮颜色。
3. 禁止删除操作没有确认弹窗。
4. 禁止异步提交按钮没有 loading。
5. 禁止权限不足的按钮仍可点击。

# 12. 弹窗 / 抽屉规范

## 12.1 弹窗类型

| 类型 | 组件 | 场景 |
|---|---|---|
| Confirm | BaseConfirm | 删除、禁用、提交确认 |
| FormDialog | BaseFormDialog | 新增、编辑 |
| DetailDialog | BaseDetailDialog | 查看详情 |
| FullscreenDialog | BaseFullscreenDialog | 大表单、大配置 |
| Drawer | BaseDrawer | 辅助信息、右侧编辑 |
| ImportDialog | BaseImportDialog | 导入数据 |
| ExportDialog | BaseExportDialog | 导出配置 |
| PreviewDialog | BasePreviewDialog | 图片、文件、报表预览 |

## 12.2 弹窗尺寸

| 尺寸 | 宽度 | 场景 |
|---|---|---|
| small | 480px | 简单确认、少量字段 |
| medium | 640px | 普通新增编辑 |
| large | 840px | 字段较多的表单 |
| xlarge | 1080px | 复杂配置 |
| fullscreen | 90vw / 90vh 或全屏 | 大表单、复杂业务 |

## 12.3 弹窗结构

```text
Dialog
├─ Header
│  ├─ Title
│  ├─ Status
│  └─ Close / Fullscreen
├─ Body
│  └─ Content / Form / Table
└─ Footer
   ├─ Cancel
   └─ Confirm
```

## 12.4 弹窗交互规则

1. 新增、编辑必须使用统一 FormDialog。
2. 弹窗标题必须明确，如“新增用户”“编辑角色”“查看详情”。
3. 弹窗底部按钮顺序：取消在左，确认在右。
4. 确认按钮异步提交时必须 loading。
5. 表单未保存时关闭弹窗，应提示确认。
6. 大表单弹窗支持最大化。
7. 弹窗内容高度超过视口时，body 滚动，footer 固定。
8. 删除、禁用、撤销必须二次确认。

## 12.5 弹窗关闭规则

表单弹窗默认：

- 禁止点击遮罩关闭。
- 允许右上角关闭，但有未保存数据时必须二次确认。
- 允许 ESC 关闭，但有未保存数据时必须二次确认。

详情弹窗可以允许点击遮罩关闭。

## 12.6 禁止事项

1. 禁止业务页面大量直接使用 `ElDialog`。
2. 禁止弹窗内再套弹窗，复杂场景改用 Drawer 或 StepDialog。
3. 禁止弹窗底部按钮顺序混乱。
4. 禁止提交后不显示 loading。
5. 禁止弹窗表单校验错误只用 message 提示。

## 12.7 其他高频组件约定

| 组件 | 约定 |
|---|---|
| Tabs | 详情页分区；过多 Tab 改左侧锚点或分组 |
| Tree | 左树右表统一 `BaseTree`，支持搜索、懒加载 |
| Steps | 流程页步骤条与底部操作栏联动，未完成步骤禁止跳过后续必填 |
| Tooltip | 表格溢出、图标按钮说明；不替代必填提示 |

# 13. 列表 / 表格规范

## 13.1 标准列表页结构

```text
BasePage
├─ BasePageHeader
├─ BaseFilterBar
├─ BaseToolbar
├─ BaseTable
└─ BasePagination
```

## 13.2 表格列顺序

```text
选择列
序号列
业务主字段
业务辅助字段
状态字段
时间字段
操作列
```

示例：

```text
[选择] [序号] [用户名称] [手机号] [角色] [状态] [创建时间] [操作]
```

## 13.3 表格操作列

1. 操作列固定在右侧。
2. 操作列宽度统一为 120 / 160 / 200。
3. 行内操作不超过 3 个。
4. 超过 3 个放入“更多”。
5. 删除、禁用等危险操作使用 danger 文案和二次确认。

## 13.4 表格状态

表格必须统一处理：

- loading
- empty
- error
- no-permission
- network-error
- partial-data

## 13.5 分页规范

1. 默认 `pageSize = 10` 或 `20`。
2. 支持 `10 / 20 / 50 / 100`。
3. 查询时重置 `page = 1`。
4. 删除最后一页最后一条后，自动回退上一页。
5. 分页统一放表格右下角。

## 13.6 禁止事项

1. 禁止页面重复封装表格 loading、分页、空状态。
2. 禁止每个页面自己写分页逻辑。
3. 禁止操作列宽度随意。
4. 禁止状态字段没有统一 tag 样式。
5. 禁止后端返回空数据时页面空白。

## 13.7 批量操作与列设置

1. 工具栏批量按钮在无选中行时必须 `disabled`。
2. 批量删除、批量启用等危险/不可逆操作必须二次确认并展示影响条数。
3. 列显示/隐藏、列宽、列顺序可提供「列设置」，偏好可存 `localStorage`（需命名空间）。
4. 服务端排序/筛选字段名与表格 `prop` 映射表应集中维护，禁止页面硬编码多套字段名。

## 13.8 表格进阶场景

| 场景 | 约定 |
|---|---|
| 服务端排序 | 传 `sortField` + `sortOrder`，与后端白名单字段一致 |
| 树形表格 | 使用 `BaseTable` 树配置或封装；懒加载子节点 |
| 固定表头 | 表格区域内部滚动，壳层不双滚动 |
| 行展开 | 展开内容用嵌套表格或描述列表，控制最大高度 |
| 可编辑单元格 | 仅明确场景使用；提交走行级或批量保存接口 |
| 合并单元格 | 仅限展示类报表，编辑类表格禁止合并 |

# 14. 筛选区规范

## 14.1 筛选类型

| 类型 | 场景 |
|---|---|
| 简单筛选 | 1 到 4 个条件 |
| 高级筛选 | 超过 4 个条件 |
| 折叠筛选 | 条件较多但不常用 |
| 侧边筛选 | 分类、标签、树结构 |
| 顶部搜索 | 关键词搜索 |
| 联动筛选 | 省市区、组织部门、业务类型 |

## 14.2 筛选区结构

```text
FilterBar
├─ KeywordInput
├─ Select
├─ DateRange
├─ AdvancedFields
└─ Actions
   ├─ Search
   └─ Reset
```

## 14.3 筛选交互规则

1. 查询按钮使用 primary。
2. 重置按钮使用 default。
3. 查询后 `page` 重置为 1。
4. 重置后清空所有条件并重新查询。
5. 日期范围必须有默认快捷项。

## 18.6 API 契约与 CodeGen（推荐）

手写 API 易导致类型与后端不一致。面向人与 AI 的研发流程中，须将契约提升为 **类型单源（Single Source of Truth，SSOT）**：**后端结构体定义 → 自动化同步 → 前端只读生成**，禁止在业务层手写与契约重复的 DTO。

### 18.6.1 契约文件与同步（Schema-Driven）

根目录（或 `contracts/`）维护由**契约网关 / OpenAPI 聚合服务**同步的 JSON（团队可命名为 `schema.json` 或按服务拆分）：

```text
contracts/
├── schema.json              # 全量或聚合契约（AI / CodeGen 优先读取）
├── openapi/
│   ├── system.yaml
│   └── business.yaml
└── README.md                # 同步命令、更新时间、负责人
```

同步脚本（CI 与本地均需可执行）：

```json
{
  "scripts": {
    "schema:sync": "node scripts/sync-schema.mjs",
    "api:gen": "pnpm schema:sync && orval --config orval.config.ts",
    "api:check": "pnpm api:gen && vue-tsc --noEmit"
  }
}
```

| 原则 | 说明 |
|---|---|
| SSOT | 业务字段以 `contracts/schema.json`（及生成的 `src/api/generated`）为准，禁止「凭感觉」造字段 |
| 人工只写薄封装 | `user.api.ts` 仅组合 generated client，不重复定义 `UserItem` 结构 |
| PR 门禁 | `api:check` 未通过禁止合并；契约变更须可见 diff |
| AI 优先读契约 | 生成 `BaseForm` / 列表列配置前，先读对应 service 在 schema 中的定义（见 §41.6） |

`schema.json` 片段示例（结构因网关而异，仅示意）：

```json
{
  "version": "2026-05-22T08:00:00Z",
  "services": {
    "system.User": {
      "create": {
        "request": {
          "username": { "type": "string", "required": true, "maxLength": 64 },
          "roleIds": { "type": "array", "items": "string" },
          "status": { "type": "enum", "ref": "UserStatus" }
        }
      },
      "page": {
        "response": {
          "records": { "type": "array", "items": "UserItem" },
          "total": { "type": "integer" }
        }
      }
    }
  }
}
```

工具链以 **OpenAPI / Swagger**（同步进 `contracts/`）为输入，生成类型与 Client：

| 场景 | 推荐工具 |
|---|---|
| 生成 TS 类型 + 请求方法 | [Orval](https://orval.dev/) |
| 仅生成类型 | openapi-typescript |
| Mock | MSW + 生成数据 |

推荐目录：

```text
src/api/
├─ generated/          # 自动生成，禁止手改
│  ├─ schema.ts
│  └─ user.ts
├─ system/
│  └─ user.api.ts      # 对 generated 的业务薄封装
└─ index.ts
```

### 18.6.2 API 版本兼容与 BFF 策略

企业级项目常见风险不是“接口不能调通”，而是前后端版本不一致导致线上页面隐性失败。API 契约须按兼容性治理：

| 变更类型 | 兼容要求 | 前端动作 |
|---|---|---|
| 新增响应字段 | 默认向后兼容 | 前端可按需使用，不得假设一定存在于旧环境 |
| 新增请求字段 | 必须有默认值或后端兼容缺省 | 前端启用前确认灰度环境已支持 |
| 字段重命名 / 删除 | 必须先标记 deprecated，保留兼容期 | generated diff 中明确影响面，禁止静默替换 |
| 枚举新增 | 前端必须有 unknown / fallback 展示 | 禁止因未知枚举导致页面报错 |
| 响应结构调整 | 视为 breaking change | 需同步版本、迁移说明与回滚方案 |

落地规则：

1. 契约变更 PR 必须包含 generated diff，Review 时关注字段删除、必填项变化、枚举变化。
2. 前端不得依赖契约中未声明字段；临时字段须在 schema 中标注实验状态和清理日期。
3. 同一业务被多端复用且接口形态差异明显时，优先通过 BFF / adapter 层收敛，而不是在页面里写分支适配。
4. BFF 层返回给前端的结构仍须进入 `contracts/schema.json`，不得成为“口头契约”。
5. 生产发布前确认前端版本、后端版本、schema 版本三者可追踪；线上问题可通过 release 与 traceId 反查对应契约。

## 18.7 禁止事项

1. 禁止在组件里直接写 axios。
2. 禁止接口返回 `any`。
3. 禁止一个 api 文件超过合理业务边界。
4. 禁止接口错误各页面单独处理。
5. 禁止请求地址散落在组件中。
6. 禁止修改 `api/generated` 下自动生成的文件（应改 OpenAPI / 契约源后重新 `schema:sync` + `api:gen`）。
7. 禁止在 `views/` 手写与 `schema.json` / generated 冲突的 interface 或字段名。

## 18.8 Mock 开发规范

| 环境 | 策略 |
|---|---|
| 本地开发 | 优先 `vite proxy` 连真实后端；无后端时用 MSW / vite-plugin-mock |
| 联调 | 使用测试环境 API，禁止把 mock 数据提交为「假接口」 |
| CI | 单测 / E2E 使用 MSW fixture，不依赖外网 |
| 生产 | **禁止** 打包 mock 中间件或假数据开关（Feature Flag 除外） |

目录建议：

```text
mock/
├── handlers/        # MSW handlers
└── data/            # 静态 JSON fixture
```

## 18.9 请求/响应字段命名转换

与后端约定二选一并在 `transform.ts` 统一实现：

| 策略 | 说明 |
|---|---|
| 全 camelCase | 拦截器出站转 snake_case，入站转 camelCase（推荐） |
| 与后端一致 | 前端类型与后端完全一致，不做转换 |

禁止在单个 api 方法内手写转换；数组、嵌套对象须深拷贝转换或经库处理。

## 18.10 Schema 驱动表单与列表（与 Base 组件）

1. 由 `schema` / generated 类型生成 `BaseForm` 的 `fields` 配置（可手写适配层 `mapSchemaToFormFields()`，**映射逻辑集中、可单测**）。
2. 表格 `columns` 同理，字段名、枚举 ref 与 schema 一致。
3. AI 提示词模板（§41.6）：「根据 `contracts/schema.json` 中 `system.User.create.request` 生成 `filterFields` / `formFields`，不得添加 schema 中不存在的字段。」

## 19.1 状态分类

| 类型 | 存放位置 |
|---|---|
| 页面临时状态 | 当前组件 |
| 表格查询状态 | useTable / 页面 |
| 弹窗状态 | useDialog |
| 用户信息 | user store |
| 权限信息 | permission store |
| 字典缓存 | dict store |
| 主题配置 | app store |
| 全局配置 | app store |
| 跨页面业务状态 | 业务 store |

## 19.2 Store 命名

```ts
useUserStore()
useAppStore()
usePermissionStore()
useDictStore()
useThemeStore()
```

## 19.3 禁止事项

1. 禁止把所有页面状态都塞进 store。
2. 禁止 store 中直接操作 DOM。
3. 禁止 store 中弹 ElMessage，除非是统一封装的业务动作。
4. 禁止长期缓存不失效的数据。

## 19.4 持久化与接口缓存

1. Pinia 持久化仅用于主题、语言、侧边栏折叠等低敏感配置，Token 优先 HttpOnly Cookie 或短期内存 + Refresh 策略。
2. 列表数据优先页内状态；跨页共享再进 store 或 `@tanstack/vue-query` 等缓存方案。
3. store 可调用 `api` 层，禁止直接写 axios。

## 19.5 Store 状态清理与重置（State Reset）

**硬性约束**：切换账号、注销、重新登录时，禁止残留上一用户的 Pinia 状态（权限菜单、用户信息、业务缓存），否则会导致越权或界面错乱。

每个全局 Store（Setup 写法）须暴露 `$reset`：

```ts
export const useUserStore = defineStore('user', () => {
  const userInfo = ref<UserItem | null>(null)
  const token = ref('')

  function $reset() {
    userInfo.value = null
    token.value = ''
  }

  return { userInfo, token, $reset }
})
```

统一登出须调用全局重置（推荐 `src/store/resetAllStores.ts`）：

```ts
import { useUserStore } from './user.store'
import { usePermissionStore } from './permission.store'
import { useDictStore } from './dict.store'

const resetFns = [
  () => useUserStore().$reset(),
  () => usePermissionStore().$reset(),
  () => useDictStore().$reset(),
]

export function resetAllStores() {
  resetFns.forEach(fn => fn())
}
```

在 `logout()` 中：**先** `resetAllStores()`，**再** 清 TagsView / 跳登录页。选项式 Store 使用 `$reset()` 或 Pinia 插件统一处理。

# 20. 路由规范

## 20.1 路由配置示例

```ts
{
  path: '/system/user',
  name: 'SystemUser',
  component: () => import('@/views/system/user/index.vue'),
  meta: {
    title: '用户管理',
    icon: 'User',

  },

```

## 20.2 Route Meta 标准

```ts
interface RouteMeta {
  title: string
  icon?: string
  permission?: string
  roles?: string[]
  keepAlive?: boolean
  hidden?: boolean
  activeMenu?: string
  breadcrumb?: boolean
  affix?: boolean          // 固定 TagsView 标签
  noCache?: boolean        // 强制不缓存
  link?: string            // 外链
}
```

## 20.3 路由守卫顺序

```text
router/
├── index.ts
├── routes.ts
├── modules/
├── guards/
│   ├── progress.guard.ts    # 顶部进度条
│   ├── auth.guard.ts        # 登录态
│   ├── permission.guard.ts  # 动态路由注册
│   └── title.guard.ts       # document.title
└── utils/
    ├── flatRoutes.ts
    └── generateMenu.ts
```

守卫顺序建议：`progress → auth → permission → title`。

## 20.4 多页签（TagsView）与 Keep-alive

1. `keepAlive: true` 的页面由壳层 `AppMain` 统一缓存。
2. 关闭标签时销毁对应缓存实例；「刷新当前页」通过 redirect 路由实现。
3. `affix: true` 用于首页等不可关闭标签。

### 20.4.1 组件 `name` 与路由 `name` 强绑定（硬性）

`<keep-alive :include="cachedNames">` 依赖组件 **`name`**，与路由 `name` 无关。规范要求：

| 项 | 要求 |
|---|---|
| 路由 | `name: 'SystemUser'`（PascalCase，与菜单/权限一致） |
| 页面组件 | `defineOptions({ name: 'SystemUser' })` |
| 一致性 | 二者**完全一致**（区分大小写），否则缓存失效 |

```vue
<!-- views/system/user/index.vue -->
<script setup lang="ts">
defineOptions({ name: 'SystemUser' })

// ... 其余逻辑
</script>
```

```ts
{
  path: '/system/user',
  name: 'SystemUser',  // 与 defineOptions.name 相同
  meta: { keepAlive: true },
  component: () => import('@/views/system/user/index.vue'),
}
```

Code Review 必查：`meta.keepAlive === true` 的页面是否声明 `defineOptions({ name })` 且与路由 `name` 一致。

## 20.5 URL 与筛选状态同步（推荐）

列表页可将筛选条件同步到 query，便于分享链接与刷新恢复，例如：

```text
/system/user?page=1&pageSize=20&username=admin&status=1
```

规则：进入页面时从 query 还原；查询/重置时更新 query；参数名与接口字段保持一致。

## 20.6 路由规则

1. 页面路由必须有 name。
2. 菜单路由必须有 title。
3. 权限路由必须配置 permission。
4. 详情页隐藏时使用 activeMenu 指向列表页。
5. 动态路由注册失败或无权访问时跳转 403，未知路径跳转 404。
6. 路由组件必须懒加载：`() => import('@/views/...')`。

# 21. 权限规范

## 21.1 权限类型

- 菜单权限
- 页面权限
- 按钮权限
- 接口权限
- 数据权限
- 字段权限
- 路由权限

## 21.2 权限编码规则

```text
模块:资源:操作
```

示例：

```text
sys:user:list
sys:user:save
sys:user:update
sys:user:delete
sys:role:assign
biz:order:export
```

## 21.3 按钮权限示例

```vue
<BaseButton
  v-permission="'sys:user:save'"
  type="primary"
>
  新增
</BaseButton>
```

或者：

```vue
<BaseButton
  v-if="hasPermission('sys:user:save')"
  type="primary"
>
  新增
</BaseButton>
```

## 21.4 权限交互规则

1. 无权限菜单不显示。
2. 无权限按钮默认不显示。
3. 特殊场景可显示但 disabled，并提供原因。
4. 403 接口错误跳转无权限页或提示。
5. 前端权限只做体验控制，后端必须二次校验。

## 21.5 权限模型（RBAC + 数据范围）

| 模型 | 含义 | 示例 |
|---|---|---|
| RBAC | 角色控制菜单与按钮 | admin、operator |
| 数据范围 DataScope | 行级数据可见范围 | 本人、本部门、本部门及下级、全部 |
| 字段权限 | 敏感字段只读或脱敏 | 手机号、身份证 |

禁止在模板写死角色判断：

```vue
<!-- 禁止 -->
<el-button v-if="user.role === 'admin'">新增</el-button>
```

数据范围过滤由后端接口保证；前端列表请求附带组织/部门参数时需与权限 store 一致。

## 21.6 权限变更刷新

用户角色或权限变更后，应重新拉取菜单与权限码，必要时刷新当前页数据，避免按钮已隐藏但接口仍可调用（后端仍须拦截）。

## 21.7 字段级权限

1. 后端返回字段可见性列表或掩码时，前端用 `v-if` / 脱敏组件渲染，禁止仅 `display:none` 仍留在 DOM（极敏感字段）。
2. 无权限字段展示 `--` 或 `***`，与 §16.3 掩码规则一致。
3. 表单编辑时，无权限字段使用 `disabled` 且不提交该字段，或提交时由后端忽略。

# 24. 错误处理规范

## 24.1 错误分类

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 网络错误 | 断网、超时 | 全局提示 + 重试 |
| 接口错误 | 500、业务异常 | 统一错误处理 |
| 参数错误 | 400 | 显示字段错误 |
| 权限错误 | 401、403 | 跳登录 / 无权限页 |
| 资源不存在 | 404 | 空状态 / 404 页 |
| 表单错误 | 必填、格式错误 | 字段下方提示 |
| 前端异常 | JS runtime error | 错误边界 + 日志上报 |
| 文件错误 | 上传失败、格式错误 | 文件项内提示 |
| 并发错误 | 重复提交 | 禁用按钮 / loading / 幂等键 |

## 24.2 错误处理原则

1. API 错误必须统一拦截。
2. 不允许业务页面到处写重复 catch。
3. 不允许直接把后端异常堆栈展示给用户。
4. 用户需要知道发生了什么、能做什么。
5. 系统需要记录错误上下文，方便排查。

## 24.3 错误类型与目录（推荐）

```text
src/utils/error/
├── AppError.ts
├── BizError.ts
├── NetworkError.ts
├── errorHandler.ts
├── errorCodeMap.ts
└── index.ts
```

| 类型 | 说明 |
|---|---|
| BizError | 业务 code 非成功 |
| NetworkError | 断网、超时 |
| AuthError | 401 未登录 |
| PermissionError | 403 无权限 |
| ValidationError | 表单/参数校验 |

## 24.4 全局异常捕获

在 `main.ts` 注册：

```ts
app.config.errorHandler = (err, instance, info) => {
  // 统一上报 + 用户友好提示
}

window.addEventListener('unhandledrejection', event => {
  // 防止 Promise 错误静默失败
})
```

组件内对关键区块使用 `onErrorCaptured`，避免整页白屏。

## 24.5 离线与服务不可用

1. 检测 `navigator.onLine`，离线时提示并限制提交。
2. 维护中、5xx 使用统一结果页（见 §36），而非无限 Toast。

## 24.5.1 表单幂等与防重复提交

1. 提交中按钮 `loading` + `disabled`。
2. 写操作可传 `Idempotency-Key`（UUID），由 request 层统一注入。
3. 支付、审批等关键流程禁止仅依赖前端防抖。

## 24.6 Axios 错误处理示例

```ts
import axios from 'axios'
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15000,
})

request.interceptors.response.use(
  response => {
    const result = response.data

    if (result.code !== 0 && result.code !== 200) {
      handleBusinessError(result)
      return Promise.reject(result)
    }

    return result.data
  },
  error => {
    handleHttpError(error)
    return Promise.reject(error)
  },
)

function handleBusinessError(error: ApiError) {
  ElMessage.error(error.message || '操作失败，请稍后重试')
}

function handleHttpError(error: unknown) {
  ElMessage.error('网络异常，请检查网络后重试')
}

export default request
```

## 24.7 错误文案规范

| 场景 | 不推荐 | 推荐 |
|---|---|---|
| 网络异常 | 请求失败 | 网络异常，请检查网络后重试 |
| 删除失败 | error | 删除失败，请稍后重试 |
| 权限不足 | 403 | 当前账号无此操作权限 |
| 表单错误 | 参数错误 | 请检查必填项和格式是否正确 |
| 文件过大 | 上传失败 | 文件大小不能超过 20MB |

# 26. 代码风格规范

## 26.1 TypeScript 规则与禁 `any` 令

1. **严禁显式 `any`**：ESLint 必须开启 `@typescript-eslint/no-explicit-any: "error"`（质量红线，CI 零 Error）。
2. 接口请求和响应必须定义类型；优先 OpenAPI CodeGen（§18.6）。
3. Props、Emits 必须定义类型。
4. 公共函数必须定义入参和返回值类型。
5. 复杂对象优先使用 `interface`；联合固定值优先 `union` 或 `enum`。
6. `tsconfig` 开启 `strict`、`noUnusedLocals`、`noImplicitAny`。
7. 提交前必须通过 `vue-tsc --noEmit`。

### 26.1.1 使用 `unknown` 替代 `any`

| 场景 | 做法 |
|---|---|
| 接口字段暂未定义 | 完善类型或 `Record<string, unknown>` |
| 第三方库无类型 | 补充 `.d.ts` 或最小包装 |
| 确实未知的运行时值 | 声明为 `unknown`，使用前 **类型守卫** 或 **断言** |

```ts
function parsePayload(raw: unknown): UserItem {
  if (!raw || typeof raw !== 'object') {
    throw new Error('Invalid payload')
  }
  const o = raw as Record<string, unknown>
  if (typeof o.id !== 'string') {
    throw new Error('Invalid id')
  }
  return { id: o.id, /* ... */ } as UserItem
}
```

禁止：`const data = res as any`、`// @ts-ignore` 掩盖类型错误（极特殊情况须 PR 说明）。

## 26.2 Vue 规则与编译期宏

1. 组件名使用多单词命名；**路由页面**须 `defineOptions({ name: 'Xxx' })`（§20.4.1）。
2. **一律**使用 `<script setup lang="ts">`。
3. `defineProps` / `defineEmits` **必须**采用**基于类型的声明**（Type-based），**禁止**运行时数组/对象写法。

```vue
<script setup lang="ts">
interface Props {
  list: UserItem[]
  loading?: boolean
}
const props = defineProps<Props>()

const emit = defineEmits<{
  success: []
  close: [id: string]
}>()
</script>
```

禁止：

```ts
defineProps(['list', 'loading'])           // 禁止
defineProps({ list: Array, loading: Boolean })  // 禁止
```

4. v-for 必须使用稳定 `key`。
5. 复杂逻辑抽到 composables。

## 26.3 禁止事项

1. 禁止超大单文件组件。
2. 禁止模板中写复杂表达式。
3. 禁止复制粘贴大量重复逻辑。
4. 禁止业务页面随意 deep 覆盖组件库样式。
5. 禁止提交未格式化代码。

## 26.4 工程化配置文件（必备）

```text
.eslintrc.cjs / eslint.config.js
.prettierrc
.stylelintrc
.editorconfig
commitlint.config.js
.lintstagedrc
```

ESLint 质量红线（须在配置中启用）：

```js
// eslint.config.js 片段
'@typescript-eslint/no-explicit-any': 'error',
'@typescript-eslint/no-unsafe-assignment': 'warn',
```

`lint-staged` 提交前至少执行：`eslint --fix`、`prettier --write`、受影响文件的 `vue-tsc`。

## 26.5 import 与路径

1. 业务代码统一使用 `@/` 别名指向 `src/`。
2. import 顺序：Vue/框架 → 第三方 → `@/` 内部 → 相对路径 → 样式。
3. 谨慎使用 `index.ts`  barrel 导出，避免循环依赖；公共包入口除外。

## 26.6 AST Lint：业务视图禁用原生 Element Plus（物理级约束）

仅靠 §41 提示词无法 100% 防止 AI 或人工写出 `<el-button>`、`import { ElTable } from 'element-plus'`。**必须在 ESLint 层对 AST 做隔离**：`views/`（及可选 `components/business/`）**禁止**直接使用 Element Plus，**只能**使用 `Base*` 封装。

与 Context Pinning 构成 **「软约束 + 硬约束」双轨**：

| 层级 | 手段 | 时机 |
|---|---|---|
| 软约束 | §41.4 阅读 Base 组件、Cursor Rules | 生成前 |
| 硬约束 | ESLint `no-restricted-imports` + `vue/no-restricted-component-names` | 保存 / CI |

### 26.6.1 禁止范围（示例清单，按项目裁剪）

在 `src/views/**` 中禁止（import 与模板均需覆盖）：

`ElButton`、`ElTable`、`ElDialog`、`ElForm`、`ElInput`、`ElSelect`、`ElPagination`、`ElMessage`（改走统一 feedback 封装）等；对应标签 `el-button`、`el-table`、`el-dialog` 等。

**允许例外**（须在 ESLint 配置中白名单并注释原因）：

- `components/base/**`：封装层内部可使用 Element Plus。
- `components/layout/**`：壳层若未封装则可单独开 override 配置块。
- `plugins/element-plus.ts`：全局注册入口。

### 26.6.2 ESLint Flat Config 示例

```js
// eslint.config.js
import pluginVue from 'eslint-plugin-vue'

const elementRestricted = {
  paths: [
    {
      name: 'element-plus',
      message: '业务视图请使用 @/components/base，禁止直接 import element-plus',
    },
  ],
  patterns: [
    {
      group: ['element-plus/*', 'element-plus/es/components/*'],
      message: '请使用 Base 组件封装',
    },
  ],
}

const restrictedElementComponents = [
  'el-button', 'el-table', 'el-dialog', 'el-form', 'el-input',
  'el-select', 'el-pagination', 'el-drawer', 'el-message-box',
]

export default [
  // ... 全局规则
  {
    files: ['src/views/**/*.{vue,ts,tsx}'],
    rules: {
      'no-restricted-imports': ['error', elementRestricted],
    },
  },
  {
    files: ['src/views/**/*.vue'],
    plugins: { vue: pluginVue },
    rules: {
      'vue/no-restricted-component-names': [
        'error',
        {
          disallow: restrictedElementComponents,
          message: '请使用 BaseButton / BaseTable 等基础组件',
        },
      ],
    },
  },
  {
    files: ['src/components/base/**/*.{vue,ts}'],
    rules: {
      'no-restricted-imports': 'off',
      'vue/no-restricted-component-names': 'off',
    },
  },
]
```

### 26.6.3 IDE 与 CI

1. 开启 ESLint **保存时自动修复/校验**（VS Code/Cursor `editor.codeActionsOnSave`）。
2. CI 中 `pnpm lint` 与 PR 门禁绑定（§39.3），**零 Error**。
3. 可选：自研 `eslint-plugin-internal` 扩展 `no-el-in-views`，在 AST 上校验 `ImportDeclaration` 与 `VElement` 节点，错误信息指向对应 `Base*` 组件文档。

## 27.5 组件生命周期治理

基础组件是企业前端的“公共 API”，须像后端接口一样治理版本、状态和兼容性。

| 状态 | 含义 | 使用规则 |
|---|---|---|
| experimental | 新增或试点组件，API 可能调整 | 仅限指定模块使用，需在 README / Storybook 标注 |
| stable | 已通过多个页面验证，API 稳定 | 推荐业务页面优先使用 |
| deprecated | 已有替代方案，进入迁移期 | 禁止新页面继续使用，旧页面按计划迁移 |

组件进入 `stable` 前必须满足：

1. Props / Emits / Slots 类型明确，并提供文档或 Storybook 示例。
2. 覆盖 loading、disabled、empty、error、readonly 等常见状态。
3. 有基础单测；复杂交互组件须补充交互测试。
4. 满足 §31 可访问性要求，图标按钮、弹窗、表单、表格类组件必须明确键盘行为。
5. 样式仅依赖 Token，不在组件内部写业务色值。
6. Breaking change 必须提供迁移指南，必要时提供 codemod 或批量替换说明。

设计 Token、Base 组件或公共交互变更时，须触发视觉回归检查；如更新基线，PR 中说明原因和影响页面。

## 27.6 Composables 约定

| Hook | 职责 | 禁止 |
|---|---|---|
| `useTable` | 分页、查询、loading、selection、刷新、删除后回退页码 | 写 DOM、直接 ElMessage（走统一错误） |
| `useDialog` | 打开/关闭、当前行、提交 loading | 内含具体业务 api 名（可传 fetcher） |
| `useRequest` | 封装单次请求状态、错误、重试 | 在内部写死业务 URL |
| `usePermission` | 权限码判断 | 缓存过期权限 |

`useTable` 推荐返回：

```ts
{
  queryForm, loading, tableData, total, selectedRows,
  fetchList, handleSearch, handleReset, handlePageChange,
}
```

列表页优先 `useTable` + `BaseTable`，避免每页复制分页逻辑。

## 27.7 图标规范

1. 统一使用 `@element-plus/icons-vue` 或项目指定 iconfont，禁止混用多套图标库。
2. 尺寸：`16px`（表格/按钮内）、`20px`（菜单）、`24px`（空状态）。
3. 颜色继承 `currentColor` 或 Token，禁止写死 `#333`。
4. 纯图标按钮必须 `aria-label`（见 §31）。

## 27.8 富文本与 JSON 编辑（可选）

| 组件 | 规范 |
|---|---|
| 富文本 | 选用带消毒的编辑器；粘贴过滤；图片走上传接口 |
| JSON | `vue-json-pretty` 或 Monaco 只读预览；编辑须校验 JSON.parse |

## 27.9 组件废弃流程

1. 标记 `@deprecated` 并在 Storybook/README 说明替代组件。
2. 至少保留 **1 个小版本** 兼容期，控制台 `warn` 一次。
3. 删除前 grep 全项目无引用，并在 CHANGELOG 记录 breaking change。

# 30. 安全规范

## 30.1 前端安全重点

- XSS 防护。
- CSRF 配合后端。
- Token 存储安全。
- 敏感信息脱敏。
- 权限绕过防护。
- 上传文件校验。
- 下载链接鉴权。
- 路由白名单。
- 依赖漏洞扫描。
- CSP（Content-Security-Policy）。
- 敏感页水印。
- 会话空闲超时登出。

## 30.2 安全工具目录（推荐）

```text
src/utils/security/
├── sanitize.ts          # XSS：DOMPurify 消毒
├── safe-html.ts         # 唯一允许渲染 HTML 的入口（内部调用 sanitize）
├── storage-crypto.ts    # localStorage 敏感字段加解密（AES 等）
├── mask.ts              # 展示脱敏
├── file-validate.ts
└── index.ts
```

## 30.3 关键规则与禁止事项

### v-html 禁用令

1. **原则上严禁**在业务模板使用原生 `v-html`（防 DOM XSS）。
2. 必须展示后端 HTML（公告、日志）时，仅允许通过 `safeHtml(content)` 或 `<SafeHtml :content="..." />`，内部调用 `sanitize.ts`（DOMPurify + 标签白名单）。
3. 禁止 `v-html="row.remark"` 直接绑定接口字段。

### 敏感数据本地缓存

1. **禁止**在 `localStorage` / `sessionStorage` **明文**存：Token、密码、手机号、身份证、工号、订单金额、内网 IP 等。
2. Token 优先 **HttpOnly Cookie**；若必须前端存储，走 `storage-crypto.ts`（如 AES，密钥勿写死在前端 bundle，政企项目宜以后端会话为准）。
3. 非敏感偏好（主题、语言、侧栏折叠）可明文存，但须命名空间前缀，如 `app:theme`。

### 其他

1. 禁止前端写死管理员权限或仅靠角色名 `admin` 判断。
2. 禁止把 token、密码拼在 URL 上。
3. 禁止控制台、日志、埋点打印 token、密码、完整手机号/身份证。
4. 上传须校验后缀、MIME、大小；禁止仅校验后缀名。
5. 生产 Source Map 不公开暴露，或上传至私有符号服务。
6. 定期 `pnpm audit`，高危漏洞阻塞发布。
7. **iframe 内嵌**：`sandbox` 限制权限；跨域通信用 `postMessage` 白名单；`meta.link` 外链须校验域名。
8. **Cookie / 合规**：涉及个人信息采集须告知用途；敏感系统启用水印（§30.1）。

## 30.4 开源与许可证

1. 新增 npm 包须确认许可证（MIT/Apache 等）与商业授权（字体、图标、图表）。
2. 禁止引入 GPL 等与闭源分发冲突的依赖（除非法务确认）。
3. 在 `NOTICE` 或 `third-party-licenses` 中汇总重大依赖（可选自动化）。

# 31. 可访问性规范

## 31.0 目标等级

企业后台建议达到 **WCAG 2.2 AA**（政企、金融、跨国团队项目建议按 AA 验收）。如因历史组件限制无法一次达成，须在待完善清单中记录豁免范围、负责人和修复日期。

## 31.1 基础要求

1. 所有可点击元素可通过键盘访问。
2. 表单项必须有关联 label。
3. 图标按钮必须有 aria-label 或 title。
4. 错误信息要能被读屏理解。
5. 颜色不能作为唯一的信息表达。
6. 焦点状态必须可见。
7. 弹窗打开后焦点进入弹窗，关闭后焦点返回触发按钮。

## 31.2 颜色对比

1. 正文文字与背景应保持足够对比度。
2. 状态颜色必须同时配合文字、图标或标签，不只靠颜色区分。

## 31.3 其他要求

1. 支持 `prefers-reduced-motion` 减少动效。
2. 异步校验、Toast 使用 `aria-live="polite"` 区域。
3. 复杂表格提供行/列摘要或操作说明，避免仅依赖颜色区分状态。

## 31.4 可访问性验收

自动化工具只能发现部分问题，核心流程必须结合人工检查：

| 场景 | 验收要求 |
|---|---|
| 键盘导航 | 仅使用键盘可完成登录、搜索、筛选、提交、关闭弹窗等核心流程 |
| 焦点管理 | Modal / Drawer 打开后焦点进入容器，关闭后回到触发元素 |
| 表单错误 | 错误信息与字段关联，读屏可感知错误原因 |
| 图标按钮 | 纯图标按钮有 `aria-label`，不可只依赖 tooltip |
| 动态内容 | Toast、异步校验、批量操作结果通过 `aria-live` 或等价方式提示 |
| 高对比 / 缩放 | 200% 缩放下核心内容不遮挡、不丢失；状态不只靠颜色表达 |

推荐在 CI 或本地检查中加入 `axe` / Playwright accessibility smoke test；但通过自动化不代表通过人工验收。

# 32. 性能规范

## 32.1 加载性能

1. 路由懒加载。

## 32.4 Web Vitals 与体积预算（建议）

| 指标 | 建议目标 |
|---|---|
| LCP | < 2.5s |
| INP | < 200ms |
| CLS | < 0.1 |

| 资源 | 建议 |
|---|---|
| 入口 chunk（gzip） | 按项目设定上限，如 main < 250KB |

# 36. 全局反馈与空状态规范

## 36.1 Message / Notification / MessageBox

| 方式 | 场景 |
|---|---|
| Message | 轻量操作结果（保存成功、删除失败） |
| Notification | 需停留较久、可带标题的系统通知 |
| MessageBox.confirm | 删除、禁用等二次确认 |
| MessageBox.alert | 强提示、不可忽略说明 |

规则：

1. 同一操作避免 Message 堆叠，可做防抖或替换。
2. 接口全局错误已在拦截器处理时，页面 catch 内不再重复 `ElMessage`。
3. 危险操作统一 `BaseConfirm`，禁止每页手写不同文案结构。

## 36.2 空状态 Empty

| 场景 | 展示 |
|---|---|
| 无数据 | 插图 + 文案 + 可选「新建」 |
| 无权限 | 说明 + 联系管理员 |
| 筛选无结果 | 建议调整筛选条件 |
| 网络错误 | 重试按钮 |

统一使用 `BaseEmpty`，禁止空白表格无提示。

## 36.3 结果页 Result

403 / 404 / 500 / 网络异常 / 系统维护 使用 `BaseResult` 或统一错误页组件，提供返回首页或重试操作。

## 36.4 骨架屏 Skeleton

首屏、表格初次加载、弹窗内容加载使用 `BaseSkeleton`，与 `v-loading` 区分：骨架用于结构占位，loading 用于局部刷新。

# 37. 日志与监控规范

## 37.1 监控目录

```text
src/monitor/
├── error-monitor.ts
├── performance-monitor.ts
├── api-monitor.ts
├── behavior-monitor.ts
└── index.ts
```

## 37.2 监控指标

| 类型 | 指标 |
|---|---|
| 性能 | FCP、LCP、CLS、INP、TTFB |
| 接口 | 耗时、失败率、HTTP 状态 |
| 错误 | 未捕获异常、Promise rejection、资源加载失败 |
| 业务 | 关键流程失败（提交、支付、审批） |
| 白屏 | 根节点是否渲染 |

## 37.3 上报方案

| 能力 | 可选方案 |
|---|---|
| JS 错误 | Sentry、Fundebug、自研 |
| 性能 | Sentry Performance、web-vitals |
| 埋点 | 自研 SDK、神策、GrowingIO |

上报字段与 §29.4 一致，须包含 `traceId`、`release` 版本号。

## 37.4 隐私

埋点须脱敏，遵守个人信息保护要求；可配置关闭行为采集的环境（如政务内网）。

## 37.5 关键业务埋点（建议）

| 事件 | 字段 |
|---|---|
| 登录成功/失败 | userId（哈希）、方式、耗时 |
| 导出 / 导入 | 模块、条数、结果 |
| 删除 / 批量操作 | 模块、条数、操作码 |
| 审批提交 | 流程 id、节点、结果 |

禁止上报密码、完整证件号、未脱敏手机号。

## 37.6 SLO 与告警

监控不是只“有数据”，还要能判断是否影响用户。生产环境建议建立前端 SLO：

| 指标 | 建议目标 | 告警建议 |
|---|---|---|
| JS 错误率 | < 0.5% sessions | 连续 10 分钟超阈值告警 |
| 白屏率 | < 0.1% sessions | 任意版本突增立即告警 |
| API 失败率 | < 1% requests（排除 4xx 业务校验） | 核心接口连续 5 分钟超阈值告警 |
| LCP | p75 < 2.5s | 新版本较上一版本劣化 20% 告警 |
| INP | p75 < 200ms | 核心页面劣化 20% 告警 |
| 核心流程成功率 | 按业务设定，如提交成功率 > 99% | 低于阈值告警并暂停灰度 |

规则：

1. 每个生产 release 必须可在监控平台按版本筛选。
2. 告警须绑定负责人或值班组，避免无人认领。
3. 新版本发布后至少观察 30 分钟核心指标，异常时停止扩大流量。
4. 故障复盘须记录影响范围、根因、恢复动作和预防措施。

## 37.7 TraceId 与版本闭环

前端请求、错误、埋点、后端日志必须能串起来：

1. 每个请求携带或透传 `traceId`；若后端生成，则前端从响应头读取并写入错误上下文。
2. 错误上报至少包含 `traceId`、`release`、`route`、`userId`（脱敏）、`tenantId`（如有）、浏览器信息和接口路径。
3. 前端构建产物须注入 `release` 版本号，与 Git commit、CI build id、sourcemap 一一对应。
4. sourcemap 上传到监控平台后不得公开暴露在 CDN；生产排查通过 release 关联 sourcemap。
5. 关键业务失败应同时上报业务事件和技术错误，便于区分“接口失败”和“用户操作失败”。

# 39. CI/CD 与质量门禁

## 39.1 流水线阶段

```text
install → lint → type-check → unit-test → build → (e2e) → deploy
```

## 39.2 GitHub Actions 示例

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main, develop]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm type-check
      - run: pnpm test
      - run: pnpm build
```

## 39.3 质量门禁

合并至 `develop` / `main` 前，流水线须满足：

| 检查项 | 门禁规则 | 未通过则 |
|---|---|---|
| ESLint | **零 Error**；须拦截 `no-explicit-any`；`views/` 禁原生 `el-*`（§26.6） | 阻塞合并 |
| schema / api:check | `pnpm api:check` 契约与类型一致（§18.6.1） | 阻塞合并 |
| vue-tsc | 全量类型检查通过 | 阻塞合并 |
| 单元测试 | 工具函数 / composable 语句覆盖率 **≥ 80%**（可调） | 阻塞合并 |
| Vite 打包体积 | 任一 chunk **gzip 前** > 500KB → Warning；> 800KB → Error（可配置） | Warning / 阻塞 |
| pnpm build | 构建成功并产出静态资源 | 阻塞合并 |
| 依赖 audit | 高危漏洞 | 阻塞发布 |

体积检查可结合 `bundlesize`、自定义脚本解析 `stats.html`，或 CI 中解析 `dist/assets/*.js` 体积。

## 39.4 发布

1. 生产构建使用锁定依赖 `frozen-lockfile`。
2. 静态资源走 CDN 时配置缓存策略；`index.html` 短缓存。
3. 每次发布记录版本号与变更日志（CHANGELOG）。
4. 遵循 **Semantic Versioning**：breaking change 升 major，功能升 minor，修复升 patch。
5. 发布前确认回滚方案，包括上一稳定版本、回滚命令、配置回退和负责人。
6. 静态资源文件名必须带 hash，CDN 长缓存；入口 HTML、运行时配置短缓存或不可缓存。
7. release 版本号必须写入前端运行时环境，并与监控、sourcemap、Git commit 关联。
8. 核心业务高峰期、财务结算窗口、重大活动期间原则上冻结非紧急发布。
9. 发布后观察 §37.6 指标，异常时优先止血：关闭 Flag、回滚入口版本或回退配置。

## 39.5 Feature Flag 与灰度（可选）

1. 开关由配置中心或环境变量注入，禁止硬编码在组件内。
2. 未开启功能的路由应不可访问（路由守卫 + 菜单隐藏）。
3. 发布失败时可关闭 Flag 回滚，无需重新构建（若后端支持动态配置）。
4. 每个 Flag 必须有 owner、创建原因、默认值、预计清理日期。
5. Flag 判断逻辑集中封装，禁止在多个页面散落读取原始配置。
6. 灰度维度可按租户、用户、角色、区域、百分比或环境划分，规则须可审计。
7. 灰度期间重点观察错误率、白屏率、核心流程成功率和接口失败率。
8. 指标异常时停止扩大流量，必要时立即关闭 Flag 或回滚版本。
9. 灰度完成后删除长期无用 Flag，避免形成永久分支逻辑。

推荐灰度节奏：

```text
内部账号 → 单租户 / 小流量 → 10% → 50% → 100%
```

每一阶段都须有明确的观察窗口和通过标准；核心业务系统不得“直接全量 + 线上试错”。

# 41. Codex / Cursor AI 编码规则

## 41.1 AI 必须遵守的规则

1. 修改代码前先理解项目结构。
2. 不允许新增重复基础组件。
3. 不允许直接绕过 BaseButton、BaseDialog、BaseTable。
4. 不允许在业务页面 hardcode 主题颜色。
5. 不允许直接在组件中写 axios。
6. 不允许忽略 TypeScript 类型。
7. **严禁**生成显式 `any`；不确定类型使用 `unknown` + 类型守卫。
8. 不允许删除权限判断。
9. 不允许删除错误处理。
10. 不允许修改公共组件 API，除非同步修改所有调用处。
11. 不允许引入新依赖，除非明确说明原因。
12. 生成代码后必须自检。
13. 单文件不超过 400 行，超过须拆分 composable 或子组件。
14. 禁止在模板内堆叠 API、权限、表格、表单、弹窗全部逻辑。

## 41.2 AI 生成标准列表页必须包含

- PageHeader
- FilterBar
- Toolbar
- Table
- Pagination
- Dialog
- Loading
- Empty
- Error
- Permission
- TypeScript 类型
- API 方法

## 41.3 AI 输出前自检

1. 是否使用了公共组件？
2. 是否符合命名规范？
3. 是否没有硬编码颜色？
4. 是否处理 loading？
5. 是否处理 error？
6. 是否处理 empty？
7. 是否处理权限？
8. 是否没有 any？
9. 是否没有重复逻辑？
10. 是否可以通过 `pnpm lint / pnpm type-check / pnpm build`？
11. 列表页是否使用 `useTable` 且含 error 态？
12. 是否未把 mock 打进生产包？
13. `keepAlive` 页面是否 `defineOptions({ name })` 与路由 `name` 一致？
14. 是否先读 `contracts/schema.json` 再写字段？
15. `views/` 是否通过 ESLint（无 `el-*`）？

## 41.4 基础组件上下文锚定（Context Pinning）

防止 AI 退化为原生 `el-table` / 虚构 `BaseTable` Props：

1. **先阅读后编写**：生成或重构 `views/` 前，须阅读 `@/components/base/` 对应组件源码，或 `.cursor/rules/10-base-components.mdc`。
2. **严禁凭空捏造**：`BaseTable`、`BaseForm`、`BaseFilterBar` 的 props / slots 必须在仓库内可稽核。
3. **提示词约束**（可写入 AGENTS.md）：「未阅读 Base 组件定义前，不得编写业务页面。」

## 41.5 AI 研效双轨约束（Context Pinning + AST Lint）

| 问题 | 仅 Prompt | 双轨后 |
|---|---|---|
| AI 使用 `el-table` | 可能仍发生 | 保存即 ESLint 报错，无法提交 |
| AI 虚构 `BaseTable` 列字段 | 靠人工 Review | schema + Base 定义双重校验 |

落地顺序建议：

1. 先沉淀 `BaseTable` / `BaseForm` 最小可用 API。
2. 配置 §26.6 ESLint 规则并接入 CI。
3. 再启用 §41.4 / §41.6 的 Cursor Rules 与 AGENTS.md。

## 41.6 Schema-Driven AI 提示词（契约锚定）

在 AGENTS.md 或 `.cursor/rules/11-schema-ssot.mdc` 中写入固定指令，例如：

```text
生成或修改 views/ 下业务页面前：
1. 读取 contracts/schema.json（或执行 pnpm schema:sync 后的最新文件）。
2. 定位当前模块对应 service（如 system.User）的 request/response 结构。
3. 据此生成 TypeScript 类型、api 调用、BaseForm fields、BaseTable columns。
4. 禁止添加 schema 中不存在的字段；禁止手写与 generated 重复的 interface。
5. UI 组件仅使用 @/components/base/*，遵守 eslint views 禁 el-* 规则。
```

对 Cursor：`@contracts/schema.json` + `@src/components/base/BaseForm` 一并加入上下文，可显著降低字段名与组件 API 幻觉。
