# Production Data Operations

生产数据修复、手工 SQL、批量回填和备份恢复属于高风险操作，必须可审批、可预演、可回滚、可审计。

## 适用范围

1. 生产库 / 预发库的手工 `INSERT`、`UPDATE`、`DELETE`、DDL、批量修复脚本。
2. 线上数据回填、状态订正、权限修复、账务 / 订单 / 用户敏感数据修复。
3. 备份、恢复、脱敏导出、从生产复制数据到非生产环境。
4. 由定时任务、一次性 Job、控制台工具触发的批量数据变更。

## 执行前

1. 必须有工单或变更记录：背景、影响租户 / 用户、Owner、Reviewer、执行人、计划窗口。
2. 必须给出 dry-run 结果：目标 SQL / 脚本、预计影响行数、抽样校验方式。
3. 必须准备回滚或前滚修复方案；不可回滚时须明确说明风险和补偿流程。
4. 涉及 PII、账务、权限、订单、库存等数据时，必须额外确认审计日志和通知策略。
5. DDL 须评估锁表、索引构建、复制延迟、磁盘增长；大表优先 expand / migrate / contract。

## SQL 与脚本约束

1. 禁止无 `WHERE` 的生产 `UPDATE` / `DELETE`；确需全表操作必须有二次审批和分批策略。
2. `WHERE` 条件须包含业务主键、租户、状态或时间窗口等边界；禁止只靠模糊条件改数据。
3. 批量脚本必须幂等，可重复执行；每批记录 traceId / jobId、影响行数、失败原因。
4. 高风险脚本先在只读查询模式输出候选记录，再切换执行模式。
5. 禁止在聊天工具中直接粘贴可执行生产密钥、连接串或完整敏感数据。

## 执行中与执行后

1. 执行时记录开始 / 结束时间、执行人、变更版本、影响行数、异常信息。
2. 执行后必须抽样校验，校验 SQL 与结果写回工单。
3. 发现影响行数与预期不一致，立即停止后续批次并通知 Owner。
4. 重要修复须补充自动化测试、迁移脚本或产品流程，避免反复手工修复。

## 备份与恢复

1. 备份须加密、设定保留周期和访问控制，遵守 `29-data-privacy-lifecycle.md`。
2. 关键系统须定期做恢复演练；不能只证明“有备份”，必须证明“能恢复”。
3. 从生产恢复到非生产环境前必须脱敏，禁止把真实 PII 进入测试 fixture。

## AI 生成约束

1. AI 可以生成数据修复 SQL / 脚本草案，但必须附带 dry-run、影响行数校验、回滚方案和人工审批提醒。
2. AI 不得把草案描述为“可直接生产执行”。
3. AI 发现用户要求绕过审批、备份、回滚或审计时，必须拒绝并给出安全流程。

样板见 `examples/scripts/data-fix-template.sql` 与 `examples/scripts/data-fix-runbook.md`。
