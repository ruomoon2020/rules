# Business Module Extension Rules

适用于基于 RuoYi-Vue-Plus、RuoYi-Cloud-Plus、ruoyi-vue-pro、JeecgBoot、lamp-cloud 等成熟后台平台新增业务功能。本文是**业务二开装配层**：不替代 `01`–`42`，只规定新增业务时如何复用平台能力。

## 默认原则

1. 新业务默认进入既有业务模块，如 `modules/{biz}`、`ruoyi-modules/{biz}`、`yudao-module-{biz}` 或项目约定 bounded context。
2. 禁止为单个业务需求直接修改 `common`、`framework`、`core`、`starter`、`system` 等公共 / 系统模块；确需改公共能力须先按 `30-ownership-adr.md` 确认 Owner、ADR、兼容与回滚。
3. 平台已有能力必须优先复用：用户、角色、菜单、按钮权限、部门、岗位、字典、参数配置、文件 / OSS、日志、审计、任务调度、租户、数据权限、代码生成。
4. 代码生成器输出只是起点；上线前必须补齐权限、数据权限、审计、幂等、校验、错误码、性能与测试。
5. 系统模块只承载平台通用能力；禁止把业务专属字段、枚举、表、流程塞进系统模块。

## 公共模块例外

只有新增或修复**平台级通用能力**时，才允许修改 `common`、`framework`、`core`、`starter`、`system`。PR 必须同时说明：

1. 该能力为何不能放在业务模块。
2. Owner、ADR、兼容策略、迁移脚本与回滚方案。
3. 对已有用户、权限、字典、日志、任务、租户、数据权限的影响。
4. 是否需要新增 eval、ArchUnit、迁移校验或回归用例。
5. 单个业务需求禁止直接修改 generator 全局模板；确需改模板，按本节处理并回归已有生成场景。

## 新增业务闭环

新增一个管理后台业务功能时，按以下顺序检查：

1. **业务边界**：确认所属模块、Owner、是否需要 ADR。
2. **命名前缀**：模块名、包名、表前缀、权限码前缀、错误码前缀、字典 type 前缀须一致。
3. **表结构**：Flyway / SQL 迁移、唯一约束、索引、逻辑删除、多租户字段、审计字段。
4. **契约**：先改 OpenAPI；DTO、分页、错误码、枚举与前端对齐。
5. **持久化**：简单 CRUD 用 MyBatis-Plus；复杂查询用 XML；排序字段白名单。
6. **服务层**：事务、幂等、状态机、数据权限、对象级授权。
7. **接口层**：权限码、参数校验、错误响应、分页结构。
8. **平台能力**：菜单 / 按钮权限、字典、操作日志、导入导出、文件、任务调度按平台既有实现接入。
9. **验证**：单元 / 集成 / 越权 / 多租户 / 导入导出 / 性能预算按场景补齐。

## 平台能力复用

| 能力 | 业务扩展要求 |
|---|---|
| 用户 / 角色 / 权限 | 复用平台 RBAC（如 system 用户、角色、菜单、部门、岗位）；禁止新增一套账号、角色、权限表 |
| 菜单 / 按钮 | 新增功能须登记菜单、按钮权限码、API 权限；后端权限注解、菜单权限码、前端按钮权限码须同源或可追踪 |
| 字典 / 参数 | 复用平台 dict / config；稳定业务状态进入平台字典或 OpenAPI 枚举，禁止前后端各写一份文案 |
| 日志 / 审计 | 复用平台 `@Log`、AOP、事件、登录 / 操作 / 错误日志、monitor；字段标准见 `09`、`27` |
| 文件 / OSS | 复用平台文件 / OSS；下载须鉴权、有效期、审计 |
| 任务 / 批处理 | 复用平台调度与任务日志；禁止手写 `new Thread` 或私有调度器 |
| 租户 / 数据权限 | 列表、详情、导出、批量、任务、异步消费都要考虑租户和数据范围；须确认注解 / 拦截器 / Mapper 条件 / 租户插件在该路径实际生效 |
| CodeGen | 复用平台 generator；生成后必须人工补充权限、审计、数据权限、索引、测试与错误码；禁止为单业务改全局模板 |

`@Log` 只代表接入平台操作日志；删除、导出、权限变更、跨租户、批量变更等高风险操作仍须满足 `27-audit-log.md` 的结构化审计字段和失败策略。

## 树表与主子表

1. 树表须校验父节点归属、租户一致性、数据权限、循环父子关系和最大层级；禁止跨租户挂载父节点。
2. 主子表须在同一事务内处理，校验主表与所有子表的对象级权限；失败时不得留下孤儿子表。
3. 主子表批量导入 / 导出须复用平台文件、审计、任务和错误明细能力。
4. CodeGen 生成的树表 / 主子表代码只能作为初稿，须补数据权限、事务边界、并发和测试。

## 命名前缀

同一业务模块须保持前缀一致，便于检索、权限治理和删除迁移：

| 项 | 示例 |
|---|---|
| 模块 / 包 | `crm`、`trade`、`wms` |
| 表前缀 | `crm_customer`、`trade_order` |
| 权限码 | `crm:customer:list`、`crm:customer:export` |
| 错误码 | `CRM_CUSTOMER_NOT_FOUND` |
| 字典 type | `crm_customer_level` |
| 事件 / 日志 action | `CRM_CUSTOMER_CREATE` |

禁止同一业务在表、权限、错误码、字典中使用多套前缀。

## 租户与系统表边界

1. 新业务表默认按项目租户模型隔离；若声明为全局表，须说明业务原因、访问权限、数据泄露风险和审计策略。
2. 系统表、租户表、字典表、权限表是平台资产；业务字段不得直接塞入这些表。
3. 需要扩展平台表时，优先使用扩展表或配置能力；确需改表须按“公共模块例外”处理。

## 禁止

- 在业务模块复制 `UserService`、`RoleService`、`DictService`、`LogService`、`FileService`、`JobService` 等公共能力。
- 为了一个业务字段修改系统用户、角色、部门、字典等平台核心表，且无 ADR 和迁移方案。
- 生成 CRUD 后直接上线，未补权限码、菜单、审计、数据权限和越权测试。
- 为单个业务需求修改 generator 全局模板，且无 Owner、ADR、兼容与回归。
- 列表加了租户条件，但详情、导出、批量删除、异步任务漏掉数据权限。
- 导出、导入、任务、回调绕过平台文件、日志、审计、任务调度能力。
- 只有菜单 / 前端按钮权限，没有后端 API 权限校验。
- 全局表未说明原因，或租户表缺少租户条件与跨租户测试。
- 树表父子关系、主子表子记录绕过租户 / 数据权限校验。
