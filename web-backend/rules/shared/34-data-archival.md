# Data Archival & Cold / Hot Tiering

与 `29-data-privacy-lifecycle.md`（保留周期）、`07-persistence-mybatis.md`（查询与索引）、`25-jobs-scheduling.md`（归档任务）互补。

## 适用场景

业务表数据量持续增长（订单、日志、审计、操作记录等），须在**不影响在线核心路径**前提下做归档与冷热分层。

## 规则

1. 归档策略须在文档中定义：触发条件（如超过 N 月）、目标存储（历史表 / 冷库 / OSS）、保留周期、物理删除时间。
2. 归档任务须 **幂等**、可断点续跑、有 jobId / 批次审计；多实例须防重（见 `25`）。
3. 在线 API 默认查询**热数据**；查询冷/归档数据须单独接口或权限，禁止无意拖慢主库。
4. 归档后 API 行为须在 OpenAPI 或产品说明中写清：历史数据如何查、是否异步导出。
5. 归档不得绕过租户、数据权限与脱敏规则。
6. 从归档恢复至在线库须走变更流程（见 `31-production-data-ops.md`）。

## 禁止

- 无索引、无分批地对亿级表做一次性 `DELETE` 或 `UPDATE`。
- 归档任务在业务高峰全表扫描。
- 归档到可公开访问的对象存储且无鉴权。
