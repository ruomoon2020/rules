# Data Access, Tenancy, and Cache Rules

用于数据权限、多租户、缓存一致性与数据脱敏。

## 数据权限 / 多租户

1. 多租户系统所有查询必须带 tenant 约束或通过统一拦截器注入，禁止业务 SQL 漏写租户条件。
2. 数据权限（本人、本部门、组织树）应在统一组件、Service helper 或 MyBatis 拦截器实现，禁止每个 Mapper 手写一套。
3. 后端必须二次校验数据权限；前端隐藏按钮或菜单不构成安全边界。
4. 跨租户管理接口必须显式权限码、审计日志和操作原因。
5. 导出、下载、详情、批量操作与列表查询使用同一数据权限口径。

## 租户配额（多租户）

1. 按租户配置：API QPS、导出条数上限、存储配额、并发任务数（按产品）。
2. 超配额返回统一 `errorCode`，并记录审计/告警；禁止静默拖垮共享资源（噪声邻居）。
3. 配额变更须审计；默认值与调优须有 Owner。

## 缓存

1. 使用缓存前必须说明 key 设计、TTL、失效策略、一致性要求。
2. 禁止缓存未脱敏敏感对象、Token、密码、完整证件号。
3. 写操作影响缓存时，必须同步删除或更新相关 key；不得只更新数据库。
4. 缓存 key 必须包含租户、用户或权限边界（若数据受其影响）。
5. 缓存穿透、击穿、雪崩风险按项目中间件方案处理。
6. `@Cacheable` 不得包住有副作用的方法。

## 数据脱敏

1. 响应 DTO、导出文件、错误明细、审计日志按权限脱敏。
2. 内部 Entity 可以保留原值，但出 API / 日志 / 导出前必须转换。
3. 脱敏规则应集中实现，禁止页面或接口各写一套。

## 禁止

- 忘记 tenant 条件的列表 / 导出 / 详情接口。
- 修改数据后不处理缓存。
- 缓存权限相关响应但 key 不含权限或租户维度。
