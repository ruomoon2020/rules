# Performance

## 必须

1. 列表必须分页；单页行数上限（如 100～500，可配置）。
2. 禁止 N+1；批量查询、JOIN 或 MP 批量接口。
3. 慢 SQL 记录（> 阈值）含 traceId。
4. 连接池大小按环境配置；禁止阻塞事件线程池。
5. 核心接口、导入导出、批处理须写明数据量上限与降级策略；超过阈值改异步任务。
6. 大表列表须说明索引、排序字段、`count` 策略；禁止在无索引字段上做深分页排序。
7. 核心接口须声明 QPS / 并发用户、P95 / P99、超时、线程池 / 连接池饱和阈值（可在 `PERFORMANCE_BUDGET.md`）。
8. 单请求最大扫描行数、返回行数、内存占用和外部依赖调用次数须有上限；超上限应异步化、分页或限流。

## 参考预算

以项目 `PERFORMANCE_BUDGET.md` 为准（可从 `docs/PERFORMANCE_BUDGET.template.md` 复制）；超预算 PR 须说明原因与优化计划。

| 指标 | 参考 |
|---|---|
| 列表接口 P95 | < 500ms（视业务） |
| 列表接口 P99 | < 1500ms（视业务） |
| 核心写接口 P95 | < 300ms（视业务） |
| 外部依赖超时 | 300ms～3s（按依赖分级） |
| 慢 SQL 阈值 | 500ms～1s（按项目） |
| 单页最大行数 | ≤ 100（无导出场景） |
| 单请求扫描行数 | 需有索引与 explain 说明 |
| 连接池 / 线程池饱和 | 需有告警阈值 |
| 导入批次 | 异步 + 分批提交 |

## 容量与压测

1. 核心接口 PR 若改变查询路径、索引、缓存、外部调用或批处理规模，须说明性能影响。
2. 导出 / 导入 / 批处理须声明最大行数、批大小、内存上限、超时和失败恢复方式。
3. 关键链路须有 P95 / P99、错误率、线程池、连接池、慢 SQL 观察点。
4. 超过预算时不得只扩大超时时间；先评估 SQL、索引、缓存、异步化和限流。
5. 压测须记录数据量、并发、QPS、响应分位、错误率、资源水位和瓶颈结论。
6. 性能预算超标合并须有 Owner、复测日期和降级 / 回滚策略。

## SQL 与索引

1. 大表查询须提供索引、排序字段、过滤字段与 `EXPLAIN` 结论。
2. 禁止无条件 count 大表；必要时使用缓存总数、近似总数或异步统计（按产品接受度）。
3. 禁止在循环中按行查详情、查字典、查权限；须批量查询或预加载。
4. 模糊查询、JSON 字段查询、函数包列查询须评估索引失效风险。

## 缓存

使用 `@Cacheable` 须说明失效策略；禁止缓存 Entity 含敏感字段未脱敏。缓存 key、TTL、租户维度与一致性见 `24-data-access-cache.md`。
