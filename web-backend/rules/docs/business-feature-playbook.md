# Business Feature Playbook

用于 AI / 开发者在成熟后台平台上新增业务功能时按步骤落地。适合 CRUD、管理后台列表、导入导出、状态流转、批处理等常见业务。

## 1. 先确认平台已有能力

新增前先回答：

1. 是否已有用户、角色、部门、岗位、租户、数据权限可复用？
2. 是否已有字典、菜单、按钮权限、操作日志、错误日志、任务调度、文件 / OSS、代码生成器？
3. 是否只是新增业务域，而不是新增平台能力？

若答案是“已有”，只接入和扩展，不重复实现。

## 2. 业务模块落地顺序

1. 业务边界：模块名、Owner、表前缀、权限前缀、是否多租户。
2. 命名前缀：模块、包、表、权限码、错误码、字典 type、日志 action 统一。
3. DB：迁移脚本、唯一索引、查询索引、逻辑删除、租户字段、状态字段。
4. OpenAPI：Request / Response / Page / ErrorCode / Enum。
5. CodeGen：可生成初始 CRUD，但必须按规则人工补齐。
6. 后端：Controller、Service、Mapper、XML、Converter、Validation。
7. 平台接入：菜单、按钮权限、字典、日志、审计、任务、文件。
8. 测试：正常、越权、跨租户、边界分页、状态流转、导入导出、性能预算。

## 3. CRUD 最小完成定义

一个业务 CRUD 至少满足：

- API 先有 OpenAPI 契约。
- 列表有分页、排序白名单、数据权限、租户条件。
- 详情 / 修改 / 删除按资源 ID 做对象级授权。
- 新增 / 修改 / 删除有权限码和操作日志。
- 删除、批量操作、状态变更等高风险操作有审计。
- DTO 不暴露 Entity；敏感字段出 API / 日志 / 导出前脱敏。
- 查询有索引策略；大表不做无条件 `%keyword%`。
- 测试覆盖无权限、跨租户、非法状态、重复提交。
- 模块名、表前缀、权限码、错误码、字典 type 前缀一致。
- 若业务表不带租户字段，已说明是全局表并补访问权限、审计和泄露风险说明。

## 4. 代码生成后的人工补齐

代码生成器不得被视为最终代码。生成后必须检查：

| 项 | 要求 |
|---|---|
| 权限 | API、菜单、按钮权限码完整 |
| 数据权限 | list、detail、export、delete、batch、job 一致 |
| 审计日志 | 删除、导入导出、配置、状态变更、批量操作必审 |
| DTO | 不返回 Entity；枚举、错误码、分页与 OpenAPI 一致 |
| SQL | 排序白名单、索引、方言登记、无 `${}` 注入 |
| 测试 | 单测 / 集成 / 越权 / 多租户 / 导入导出 |

生成的菜单、权限码、字典、查询条件和字段类型只是初稿，必须与 OpenAPI、DB 约束、租户 / 数据权限和审计规则复核。

单个业务需求不得直接修改 generator 全局模板。确需调整模板时，须按 `43` 的公共模块例外处理，并回归普通 CRUD、树表、主子表等已有生成场景。

## 5. 树表 / 主子表补充检查

| 类型 | 必查项 |
|---|---|
| 树表 | 父节点存在、父子租户一致、父节点数据权限、循环父子、最大层级 |
| 主子表 | 主表与子表同事务、子表归属校验、失败回滚、孤儿数据清理 |
| 导入导出 | 主子关系错误明细、任务日志、下载鉴权、审计 |
| 测试 | 跨租户父节点、无权限子表、重复提交、并发修改 |

## 6. PR 自检

- 是否修改了 `common` / `framework` / `core` / `system`？若是，是否有 Owner 和 ADR？
- 是否重复实现了平台已有用户、权限、字典、日志、文件、任务能力？
- 是否新增了菜单、按钮权限、字典、日志审计？
- 后端权限注解、菜单权限码、前端按钮权限码是否一致或可追踪？
- 模块 / 表 / 权限 / 错误码 / 字典前缀是否统一？
- 是否所有入口都覆盖租户 / 数据权限，而不是只有列表？
- 数据权限注解 / 拦截器 / Mapper 条件 / 租户插件是否在当前查询、导出、任务路径实际生效？
- `@Log` 是否只是操作日志？高风险操作是否另按 `27` 满足结构化审计字段与失败策略？
- 树表 / 主子表是否补父子归属、事务、循环关系和跨租户测试？
- 是否导入导出、任务、异步消费也遵守权限、幂等、审计与 traceId？
- 若修改 `system` / `common`，是否证明这是平台能力而非业务能力，并补 ADR、迁移、回滚？
